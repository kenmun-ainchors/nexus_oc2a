
## L-171 — CREST v1.3 external-loop discipline works when Yoda enforces phase separation and independent verification
**Date:** 2026-06-24
**Source:** Ken observed TKT-0761 execution and praised the verify-error-replan-sendback(Forge)-verify-judge(Sage) discipline as the first time CREST v1.3 ran as intended. The pattern emerged because Yoda did not collapse Execute and Verify into one subagent, did not trust Forge's self-reported "Verify complete," and treated Sage's workspace-isolation failure as a blocker requiring a structural fix (TKT-0739).
**Lesson:** The CREST v1.3 value is in the loop, not the labels. Plan/Execute/Verify/Replan/Synthesize/Done only delivers quality when: (1) the orchestrator owns phase transitions, (2) the verifier is independent from the builder, (3) evidence is assembled before a verdict is rendered, and (4) a Verify failure automatically triggers Replan and re-Execute rather than being glossed over. The discipline is fragile because it is slower than self-greening; without structural guardrails, time pressure will collapse it.
**Fix:** (1) Already demonstrated TKT-0761 pattern successfully. (2) TKT-0762/TKT-0763 resolved Sage workspace isolation so Sage can act as Judge. (3) Need structural enforcement so this pattern becomes the default, not a one-off.
**Evidence:** TKT-0761: Forge Execute claimed cache parity working, but Yoda spot-check found `state/next-ticket.json` still held TKT-0530/active-sprint-ready while stdout returned TKT-0721/tracker-override. Sent back for Replan atom 7. Re-verified 9/9 parent-session checks and 5/5 regression tests. Sage Verify attempt failed due to workspace-qa sandbox isolation; Yoda pivoted to parent-session verification per L-168 and still rendered an evidence-based PASS.
**Prevention:** (1) Every Execute dispatch must include a verifier corpus and expected evidence. (2) Every Verify phase must either come from Sage/QA or have a documented exception with parent-session verification. (3) Any self-reported "Verify complete" from the executing agent must be spot-checked independently. (4) A Verify failure must produce a Replan ticket/atom, not a narrative workaround. (5) Add script/tool support so the disciplined path is faster than the shortcut.

## L-170 — Transparent canonical features must be verified at the user-facing entrypoint, not only at the new component
**Date:** 2026-06-24
**Source:** TKT-0761. CRESTv2-P1 tracker override (CHG-0759) was designed to make `db-sprint.sh next-ticket` transparently apply `state/crestv2-p1-tracker.json` `locked_execution_order`. Forge implemented it as a separate wrapper `scripts/next-ticket-tracker-override.sh`. Verification proved the wrapper returned TKT-0721, but the canonical `db-sprint.sh next-ticket --agent yoda` still returned TKT-0530. The user-facing interface therefore failed the original intent.
**Lesson:** When a feature is specified as "transparent" or "canonical" (no new call path, same entrypoint), verification must exercise the default entrypoint, not just the new helper. A component that works in isolation but leaves the canonical path unchanged is a misimplementation, not a success. CREST v2.0 execution control can enforce atom order and gates, but it cannot catch a wrong interpretation of the design intent.
**Fix:** (1) Opened TKT-0761 to move the tracker-override logic into `db-sprint.sh next-ticket` and delete/deprecate the wrapper. (2) Verifier corpus must include a black-box test: calling `db-sprint.sh next-ticket --agent yoda` directly returns TKT-0721 when the tracker is locked and TKT-0721 is eligible, and falls back to TKT-0530 only when no tracker ticket applies.
**Evidence:** `db-sprint.sh next-ticket --agent yoda` → `{"ticket":"TKT-0530","reason":"active-sprint-ready"}`. `scripts/next-ticket-tracker-override.sh --agent yoda` → `{"ticket":"TKT-0721","reason":"tracker-override"}`. `state/crestv2-p1-tracker.json` shows `in_progress_tickets: ["TKT-0721"]`, `locked_execution_order` seq 1 lists TKT-0721 after TKT-0390 (done).
**Prevention:** (1) For any "transparent" feature, require an acceptance test that calls only the original public interface. (2) Verdict must state: "original intent is observable at default entrypoint." (3) Do not allow the implementation to introduce a new wrapper/script as the primary path when the design says the existing path absorbs the behavior. (4) Sage must flag divergence between design intent (entrypoint unchanged) and implementation (new wrapper required).

## L-169 — Yoda must use canonical next-ticket resolver and never infer sprint state from raw sequence or stale memory
**Date:** 2026-06-24
**Source:** Ken test after Yoda incorrectly answered "what's the next ticket on Sprint 9?" Yoda defaulted to raw Sprint 9 sequence (TKT-0530), ignored the actual in-progress execution thread (TKT-0721 in Sprint 10), and falsely claimed Sprint 9 had ended when it ends 2026-06-29. Yoda also failed to recall the previous day's CRESTv2-P1 tracker lock and CHG-0752 rebalance decisions.
**Lesson:** "Next ticket" is a canonical platform query, not a memory or sequence inference. The pg-sprint-backlog skill package has no deterministic resolver and `db-sprint.sh current` / `db-ticket.sh list --sprint-current` disagree on the active sprint when no sprint has `status='in_progress'`. Yoda must not answer sprint/next-ticket questions without calling a canonical resolver that considers date windows, in-progress state, cross-sprint priority, and locked execution order.
**Fix:** (1) Atlas+Thrawn designed structural fix TKT-0728 / CHG-0758: add `db-sprint.sh next-ticket [--agent]`, date-window awareness in `get_current_sprint_name()`, `db-sprint.sh activate` for committed→in_progress transitions, and `state/next-ticket.json` for bootstrap injection. (2) Yoda will call `bash scripts/db-sprint.sh next-ticket --agent yoda` before answering any next-ticket or sprint-state question once the fix is deployed. (3) Until deployed, Yoda must read `state/crestv2-p1-tracker.json` and run `memory_search` for recent sprint decisions before answering.
**Evidence:** `db-sprint.sh current` returned Sprint 10; `db-ticket.sh list --sprint-current` returned Sprint 11; raw `state_sprints` query returned Sprint 9. Sprint 9 date window (2026-06-22 to 2026-06-28) contains 2026-06-24. Tracker `in_progress_tickets` shows `["TKT-0721"]`.
**Prevention:** (1) Never answer "next ticket" from raw sprint sequence. (2) Always consult canonical resolver or tracker + memory when resolver is unavailable. (3) Log a checkpoint before answering: current active sprint, next committed sprint, in-progress tickets, reason. (4) Run `memory_search` for any question involving sprint, priority, sequence, or tracker.

## L-168 — Instrumentation edits must define path variables before referencing them; subagent verification may run in the wrong workspace
**Date:** 2026-06-23
**Source:** TKT-0390 build verification. Forge inserted `DECISION_SCRIPT="$SCRIPT_DIR/pg-write-decision.sh"` and an `emit_decision()` function near the top of `scripts/check-session-model.sh`, but the existing `SCRIPT_DIR` assignment was further down. With `set -euo pipefail`, the script crashed at startup with `SCRIPT_DIR: unbound variable`. Separately, a QA verifier subagent ran in an isolated `workspace-qa` sandbox and could not see the parent workspace files, returning `UNABLE TO VERIFY`.
**Lesson:** Adding helper-variable references at the top of a shell script is unsafe if the variable is not assigned before that point — especially under nounset. Also, cross-agent subagents do not automatically share the parent workspace; verification of parent files/scripts must either run in the parent session or be explicitly dispatched to an agent with parent workspace access.
**Fix:** (1) Dispatched Forge hotfix to move `SCRIPT_DIR` assignment to line 14 of `scripts/check-session-model.sh`, before `DECISION_SCRIPT`. (2) Re-ran the session-model regression test and full episodic suite in the main session; all pass. (3) For future instrumentation tasks, require the subagent to return a parent-session smoke-test command and a verification command that Yoda runs directly, rather than relying only on a same-name subagent verifier.
**Evidence:** `check-session-model.sh --json` now returns `{"status":"ok"}`; `tests/regression/episodic/test-session-model-emits-event.sh` 3/3 PASS; full test-runner 6/6 PASS; hash chain 0 orphans.
**Prevention:** (1) Always inspect the first 30 lines of a script after instrumentation to confirm helper variables are defined before use. (2) When dispatching verification, specify `cwd` and the exact parent workspace path, or run the verifier in the main session. (3) Treat a verifier subagent's `UNABLE TO VERIFY` as a blocker, not a soft fail; pivot to parent-side verification. (4) Add a regression test that exercises the actual script entry point (not just functions) to catch startup-time variable errors.
