**Date:** 2026-06-27
**Source:** TKT-0747 mechanical cleanup of `docs/platform-lessons-register-v1.0.md`. The Python `ensure_fields()` parser used the regex `^-\s+\*\*([^*]+)\*\*:\s*(.*)$` to extract Markdown bold-field lines such as `- **Source:** memory/CHANGELOG.md#CHG-0411`. Because `([^*]+)` is greedy and `:` is not `*`, the group consumed `Source:` including the trailing colon, leaving nothing for the closing `\*\*` to match. The regex silently failed on every field line, so the parser saw an empty body, inserted `OPEN` placeholders, and appended the original body unchanged — producing duplicated/placeholder sections across the whole register.
**Lesson:** A parser regex that looks correct can silently degrade to no-op and produce plausible-looking garbage. Greedy negated character classes are especially dangerous when the delimiter-adjacent character (here the colon) is not excluded. The failure mode is not an exception; it is a data-quality regression that only shows up on careful spot-checking.
**Fix:** Changed the field-name group to exclude both `*` and `:` (or use a non-greedy form): `^-\s+\*\*([^:*]+?)\*\*:\s*(.*)$`. Re-ran the cleanup from HEAD after reverting the corrupted output. Verified zero duplicate headings, zero OPEN placeholders, and spot-checked CHG-0411/CHG-0608/CHG-0706 for correct field extraction.
**Evidence:** `docs/platform-lessons-register-v1.0.md` post-cleanup: 191KB, 3,598 lines, 162 entries; `grep -c '**Source:** OPEN'` returns 0; `grep -E '^### .*\.\.\.'` returns empty; no duplicate IDs in `^### (L-[0-9]+|...)` pattern.
**Prevention:** (1) When writing field-parsing regexes, exclude delimiter-adjacent punctuation from negated classes or use non-greedy quantifiers. (2) Always include a negative test: parser must reject/reject malformed input, and a parser that accepts 100% of inputs with no change is suspicious. (3) For document transformations, verify by sampling sections before and after, not just by absence of errors. (4) Add a regression test for `ensure_fields()` against the actual Markdown field format before reusing the script.

## L-170 — Transparent canonical features must be verified at the user-facing entrypoint, not only at the new component
**Date:** 2026-06-24
**Source:** TKT-0761. CRESTv2-P1 tracker override (CHG-0759) was designed to make `db-sprint.sh next-ticket` transparently apply `state/crestv2-p1-tracker.json` `locked_execution_order`. Verification proved the wrapper returned TKT-0721, but the canonical `db-sprint.sh next-ticket --agent yoda` still returned TKT-0530. The user-facing interface therefore failed the original intent.
**Lesson:** When a feature is specified as "transparent" or "canonical" (no new call path, same entrypoint), verification must exercise the default entrypoint, not just the new helper. A component that works in isolation but leaves the canonical path unchanged is a misimplementation, not a success. CREST v2.0 execution control can enforce atom order and gates, but it cannot catch a wrong interpretation of the design intent.
**Fix:** (1) Opened TKT-0761 to move the tracker-override logic into `db-sprint.sh next-ticket` and delete/deprecate the wrapper. (2) Verifier corpus must include a black-box test: calling `db-sprint.sh next-ticket --agent yoda` directly returns TKT-0721 when the tracker is locked and TKT-0721 is eligible, and falls back to TKT-0530 only when no tracker ticket applies.
**Evidence:** `db-sprint.sh next-ticket --agent yoda` → `{"ticket":"TKT-0530","reason":"active-sprint-ready"}`. `scripts/next-ticket-tracker-override.sh --agent yoda` → `{"ticket":"TKT-0721","reason":"tracker-override"}`. `state/crestv2-p1-tracker.json` shows `in_progress_tickets: ["TKT-0721"]`, `locked_execution_order` seq 1 lists TKT-0721 after TKT-0390 (done).
**Prevention:** (1) For any "transparent" feature, require an acceptance test that calls only the original public interface. (2) Verdict must state: "original intent is observable at default entrypoint." (3) Do not allow the implementation to introduce a new wrapper/script as the primary path when the design says the existing path absorbs the behavior. (4) Sage must flag divergence between design intent (entrypoint unchanged) and implementation (new wrapper required).

## L-173 — Never execute unreviewed shell commands; command strings must be inspected before exec
**Date:** 2026-06-27
**Source:** Yoda accidentally sent a bash fork-bomb pattern (`:(){ :|:& };:`) to `exec` while attempting to inspect process state during TKT-0344 verification. The command was not reviewed; it ran and spawned thousands of bash processes, causing load average to spike to ~690 on OC1.
**Lesson:** The `exec` tool is powerful and dangerous. Any shell command that contains recursion, backgrounding, loops, subshells, or unusual punctuation must be manually inspected before execution. A command that looks like a one-liner can be a denial-of-service attack against the host. The orchestrator must never paste or construct shell commands without reading every token.
**Fix:** (1) Killed the fork-bomb session immediately (`process kill`). (2) Verified no persistent fork-bomb processes remain; load returned to normal (~3.4). (3) No further shell-based process inspection during this incident; used `ps aux` with static, reviewed filters instead. (4) Logging this lesson in LESSONS.md.
**Evidence:** `uptime` after recovery: load averages 3.46, 55.73, 292.36 and falling. `ps aux` found no `:(){ :|:& };:` processes. PIDs 10852/11111 confirmed as idle nexus-controller-runner bash processes, not fork-bomb survivors.
**Prevention:** (1) Before every `exec`, read the command string character-by-character. (2) Reject any command with function definitions, recursive calls, `&` backgrounding, or pipe chains inside functions. (3) For process inspection, use only static commands like `ps aux | grep <literal string>` and `uptime`. (4) If unsure, ask the user or use `process list` instead of constructing shell commands. (5) Add this rule to Yoda's CREST Plan checklist: "No unreviewed shell commands in Verify phase."
