
## L-169 — Yoda must use canonical next-ticket resolver and never infer sprint state from raw sequence or stale memory
**Date:** 2026-06-24
**Source:** Ken test after Yoda incorrectly answered "what's the next ticket on Sprint 9?" Yoda defaulted to raw Sprint 9 sequence (TKT-0530), ignored the actual in-progress execution thread (TKT-0721 in Sprint 10), and falsely claimed Sprint 9 had ended when it ends 2026-06-29. Yoda also failed to recall the previous day's CRESTv2-P1 tracker lock and CHG-0752 rebalance decisions.
**Lesson:** "Next ticket" is a canonical platform query, not a memory or sequence inference. The pg-sprint-backlog skill package has no deterministic resolver and `db-sprint.sh current` / `db-ticket.sh list --sprint-current` disagree on the active sprint when no sprint has `status='in_progress'`. Yoda must not answer sprint/next-ticket questions without calling a canonical resolver that considers date windows, in-progress state, cross-sprint priority, and locked execution order.
**Fix:** (1) Atlas+Thrawn designed structural fix TKT-0728 / CHG-0758: add `db-sprint.sh next-ticket [--agent]`, date-window awareness in `get_current_sprint_name()`, `db-sprint.sh activate` for committed→in_progress transitions, and `state/next-ticket.json` for bootstrap injection. (2) Yoda will call `bash scripts/db-sprint.sh next-ticket --agent yoda` before answering any next-ticket or sprint-state question once the fix is deployed. (3) Until deployed, Yoda must read `state/crestv2-p1-tracker.json` and run `memory_search` for recent sprint decisions before answering.
**Evidence:** `db-sprint.sh current` returned Sprint 10; `db-ticket.sh list --sprint-current` returned Sprint 11; raw `state_sprints` query returned Sprint 9. Sprint 9 date window (2026-06-22 to 2026-06-28) contains 2026-06-24. Tracker `in_progress_tickets` shows `["TKT-0721"]`.
**Prevention:** (1) Never answer "next ticket" from raw sprint sequence. (2) Always consult canonical resolver or tracker + memory when resolver is unavailable. (3) Log a checkpoint before answering: current active sprint, next committed sprint, in-progress tickets, reason. (4) Run `memory_search` for any question involving sprint, priority, sequence, or tracker.

## L-168 — Instrumentation edits must define path variables before referencing them; subagent verification may run in the wrong workspace
**Date:** 2026-06-23
**Source:** TKT-0390 build verification. Forge inserted `DECISION_SCRIPT="$SCRIPT_DIR/pg-write-decision.sh"` and an `emit_decision()` function near the top of `scripts/check-session-model.sh`, but the existing `SCRIPT_DIR` assignment was further down. With `set -euo pipefail`, the script crashed at startup with `SCRIPT_DIR: unbound variable`. Separately, a QA verifier subagent ran in an isolated `workspace-qa` sandbox and could not see the parent workspace files, returning `UNABLE TO VERIFY`.
**Lesson:** Adding helper-variable references at the top of a shell script is unsafe if the variable is not assigned before that point — especially under nounset. Also, cross-agent subagents do not automatically share the parent workspace; verification of parent files/scripts must either run in the parent session or be explicitly dispatched to an agent with parent workspace access.
**Fix:** (1) Dispatched Forge hotfix to move `SCRIPT_DIR` assignment to line 14 of `scripts/check-session-model.sh`, before `DECISION_SCRIPT`. (2) Re-ran the session-model regression test and full episodic suite in the main session; all pass. (3) For future instrumentation tasks, require the subagent to return a parent-session smoke-test command and a verification command that Yoda runs directly, rather than relying only on a same-name subagent verifier.
**Evidence:** `check-session-model.sh --json` now returns `{"status":"ok"}`; `tests/regression/episodic/test-session-model-emits-event.sh` 3/3 PASS; full test-runner 6/6 PASS; hash chain 0 orphans.
**Prevention:** (1) Always inspect the first 30 lines of a script after instrumentation to confirm helper variables are defined before use. (2) When dispatching verification, specify `cwd` and the exact parent workspace path, or run the verifier in the main session. (3) Treat a verifier subagent's `UNABLE TO VERIFY` as a blocker, not a soft fail; pivot to parent-side verification. (4) Add a regression test that exercises the actual script entry point (not just functions) to catch startup-time variable errors.
