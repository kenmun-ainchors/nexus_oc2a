### Model Optimisation Strategy — Gemma4 Benchmark (TKT-0013)
**Context:** Ken wants Yoda + Aria to delegate low-value/deterministic tasks to Gemma4 sub-agents instead of using Sonnet for everything. Three questions raised and answered:

**Q1 — Keep 26b alive at 16 GB?** → NO. 13 GB baseline + 16 GB = 29 GB on 24 GB machine = swap territory. Degrades everything.

**Q2 — Need 26b for low-level tasks?** → NO. Smaller variants exist.

**Q3 — 32 GB Mac changes things?** → YES significantly. Could keep e4b permanent + load 26b on-demand without swap risk. Revisit when hardware arrives.

**Benchmark executed:**
- Script: `scripts/gemma4-benchmark.sh`
- Results: `state/benchmark/`
- Models tested: `gemma4:e2b` (5.1B, 9.6 GB?) vs `gemma4:e4b` (8.0B)
- 8 tasks: cost summary, health check, CHANGELOG format, JSON extraction, routing classify, ticket JSON, backlog filter, warm latency repeat

**Results:**
| | e2b | e4b |
|---|---|---|
| Pass rate | 4/6 (67%) | 4/6 (67%) |
| Avg latency | 6,081ms | 9,210ms |
| Failures | T02, T04 (single-word tasks) | T02, T04 (same) |

**Root cause of failures:** Both models have thinking mode enabled by default — produce verbose reasoning instead of single-word outputs. Fix: prepend `"Answer directly. One word/line only. No thinking."` to deterministic task prompts.

**Decision pending Ken approval:** Lock in `gemma4:e2b` as standard
- 34% faster than e4b
- ~2 GB less RAM
- Same quality on all tested tasks
- Update model-policy.json, cron jobs, Warden baseline to e2b
- Delete gemma4:e4b (no benefit, higher cost)
