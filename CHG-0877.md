## Problem

Ken and Angie coordinate with several personal contacts who are **not on Telegram** (notably Damo, iPhone user). The current OpenClaw agent stack on OC2A is Telegram-only for outbound personal messaging, which leaves a coverage gap for personal coordination. iMessage is the dominant channel for Australian iPhone users.

Building an iMessage bridge from a Mac Mini is feasible (AppleScript → Messages.app) but carries:
- **Apple ToS gray zone** (unsupported automation surface; Apple can change the AppleScript dictionary at any macOS release).
- **Account risk** on the Apple ID (rate-limiting, flagging for suspicious automation).
- **Privacy / security risk** (anyone with shell access on OC2A can send an iMessage appearing to come from Ken).
- **UI side effects** on the live Mac session (AppleScript may surface Messages.app).

A *naïve* build (Yoda has `osascript`, just run it) would be unsafe. A *correct* build needs: per-message human approval, recipient allowlist, daily cap, audit log, and an instant kill switch.

## Changes

This CHG documents the **prototype**, not a production rollout. All artifacts below are written, ready, and inert (no Messages.app interaction has occurred). The first real send requires explicit per-instance Ken approval.

### 1. `scripts/imessage-bridge.sh` — the bridge (one-shot, ~250 LOC, zsh)

A single CLI that takes `--to <address> --body "<text>"`, validates against a hard allowlist, writes an audit row, then invokes `osascript` with the appropriate AppleScript. **Explicit refusal paths:**

| Gate | Exit code | Why |
|---|---|---|
| Kill switch (`state/imessage-bridge.disabled` exists) | 3 | Ken can disable instantly without code changes. |
| Recipient not in allowlist | 4 | Hard allowlist; no auto-add. |
| Body length > `BODY_MAX` (default 1000) | 5 | Cap on message size. |
| Daily cap reached (default 20) | 6 | Volume protection. |
| Cannot address Messages.app (TCC denial) | 7 | Surface the Automation permission issue. |
| AppleScript runtime error | 8 | Forensics captured in audit row. |
| `--yes-after-prompt` not given | 9 | The **single human-approval gate** in the script itself. |

In MVP mode, `--yes-after-prompt` is **the only** flag that allows the script past the approval gate. Telegram approval buttons would translate to the same flag from a different code path (TBD in §3).

Supports `--dry-run` (renders the exact AppleScript, no send) and `--self-test` (probes AppleScript reachability, no send). Both are safe to run any time.

### 2. `scripts/imessage-bridge-test.sh` — pre-flight

Checks: `osascript` ↔ Messages.app reachability, allowlist presence, kill switch status, `jq` availability, config presence, today's send count. Exits 0 on green, 1 on any red. Designed to be run as a sanity check before any send.

### 3. `state/imessage-bridge.allowlist` — recipient allowlist

Plain text, one address per line. E.164 preferred (`+61412345678`); iCloud emails also accepted. **Empty by default.** No wildcards. The script greps for exact line match.

### 4. `state/imessage-bridge.config` — env-style tunables

`DAILY_CAP=20`, `BODY_MAX=1000`, `REQUIRE_CONFIRM=1`. Defaults are conservative; Ken can edit without touching the script.

### 5. `state/imessage-audit.jsonl` — fallback audit log

PG preferred (`state.imessage_sends` table, schema in `ARCHITECTURE.md` §5) but a JSONL fallback ships with the prototype for resilience. **The body is never stored** — only `body_sha256`, `body_length`, `to`, `approver`, timestamps, exit code, and the exact AppleScript run (for forensics). This is a tamper-evident receipt without storing PII.

### 6. `projects/imessage-bridge/FEASIBILITY.md` and `projects/imessage-bridge/ARCHITECTURE.md`

- **FEASIBILITY.md**: full risk register (R1–R7), recommendation, open questions (Q1–Q5) for Ken.
- **ARCHITECTURE.md**: components, sequence diagram, failure handling, audit schema.

### 7. `projects/imessage-bridge/README.md`

Operator-facing usage: setup, first-time Automation permission grant, daily operation, kill switch, troubleshooting.

## Why

iMessage coverage closes a real personal-coordination gap. The MVP is intentionally **narrow**:
- Outbound only.
- Text only, ≤ 1000 chars.
- One recipient per send.
- No inbox read, no attachments, no groups, no daemon.
- Per-message human approval enforced at the script level.
- Allowlist + daily cap + kill switch + audit log.
- Body content never stored, only hashed.

These constraints make the bridge small enough to reason about, killable in one touch, and recoverable if Apple's automation surface changes or if the Apple ID is ever flagged.

## Rollback

- **Instant kill:** `touch state/imessage-bridge.disabled` — bridge refuses all sends, no code change.
- **Full rollback:** `rm scripts/imessage-bridge.sh scripts/imessage-bridge-test.sh state/imessage-bridge.allowlist state/imessage-bridge.config` (audit log is left for forensics).
- **Apple ID risk:** if Apple flags the account, Ken re-authenticates interactively; the bridge has no Apple ID credentials of its own (it uses the live macOS Messages.app session) — so revocation is simply closing the Messages.app session.
- **No state changes outside the listed files. No cron jobs, no daemons, no PG schema changes in this CHG.**
