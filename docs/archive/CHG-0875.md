# CHG-0875: LinkedIn Token Health Probe — Auto-Refresh on 401 + Enriched Diagnostics

**Status:** ✅ Implemented & Verified
**Date:** 2026-07-13
**Author:** Yoda 🟢 (Forge execute)
**Related:** TKT-0743, CHG-0865

## Problem

The business LinkedIn token kept returning HTTP 401 / serviceErrorCode 65601, but the health probe only discovered it at the daily 07:00 cron. No auto-refresh was attempted — the probe immediately marked `status=revoked` without trying to use the stored refresh token.

## Changes

### 1. `scripts/linkedin-token-health.sh`

- **`probe_account()`**: On HTTP 401 from `/v2/userinfo`, if a refresh token exists (`refreshTokenPresent=true`), attempt token refresh by calling `zsh scripts/linkedin-auth.sh --account <acct> --refresh`.
- **Refresh outcomes**:
  - Refresh succeeds + re-probe OK → `status=ok, reason=token_refreshed`
  - Refresh succeeds but re-probe fails → `status=revoked, reason=token_refreshed_but_still_401`
  - Refresh fails with `invalid_grant` → `status=revoked, reason=refresh_token_invalid_grant, nextAction=reauth_now`
  - Refresh fails for other reasons → `status=revoked, reason=refresh_failed, nextAction=reauth_now`
  - No refresh token → falls through to existing 401 classification
- **`write_health_file()`**: Now accepts 13 parameters (added `refreshAttempted`, `refreshOutcome`, `refreshError`). State file entries enriched with these fields.

### 2. `scripts/linkedin-token-health-alert.sh`

- **Distinct alert messages**:
  - `refresh_token_invalid_grant` → 🚨 CRITICAL: full browser reauth required immediately
  - `refresh_failed` → 🔴 reauth now
  - `token_refreshed` → 🟡 auto-refreshed, no action needed
  - `token_expired` → 🔴 reauth now
  - `no_refresh_token` → 🟡 reauth soon
  - `expires_within_*` → 🟡 reauth soon
- **Bugfix**: Removed invalid `local` keyword usage outside functions (zsh restriction)
- **Bugfix**: Switched `echo "$latest"` to `print -r -- "$latest"` for unicode-safe piping

## Verification

- `zsh scripts/linkedin-token-health.sh --all --dry-run` — all 3 accounts pass
- `zsh scripts/linkedin-token-health.sh --account ken` — ✅ OK
- `zsh scripts/linkedin-token-health.sh --account angie` — ✅ OK
- `zsh scripts/linkedin-token-health.sh --account business` — HTTP 401 → refresh → `invalid_grant` → revoked with enriched state
- `zsh scripts/linkedin-token-health-alert.sh --dry-run` — 🚨 critical alert for business account
- State file correctly records: `refreshAttempted: true`, `refreshOutcome: "invalid_grant"`, `reason: "refresh_token_invalid_grant"`

## Impact

The business `invalid_grant` issue (HTTP 401/65601) would now be detected and alerted immediately on the first probe, with a clear 🚨 critical alert demanding full browser reauth. Previously it silently marked revoked and waited for the next cron cycle.

## Next Steps

- Business account needs full browser reauth: `zsh scripts/linkedin-auth.sh --account business`
- Consider adding a proactive refresh cron (e.g., weekly) to prevent expiry-related 401s
