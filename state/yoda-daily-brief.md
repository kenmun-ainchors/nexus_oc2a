### 5. Token Cost Optimisation — Platform Principle Locked
We discovered Warden was spending ~17,000 tokens every 15 minutes just to run two bash scripts — no reasoning needed at all. Fixed it to ~11,000, but more importantly, Ken and Yoda locked a set of non-negotiable platform rules:

- If a cron doesn't need AI reasoning, use a shell script — no LLM
- Always set `lightContext: true` on background crons (stops them loading the full platform memory on every run)
- Scripts write results to JSON files; the AI reads the summary — never have a script dump hundreds of lines back into the AI's context
- Match the model to the task: tiny checks → Haiku, content writing → Sonnet, architecture → Sonnet or Opus

These rules are now in RULES.md permanently.

### 7. Specialist Agent Delivery Pattern — Lesson Learned
We hit a problem: when work was delegated to Atlas via sessions_send, it went to Atlas's heartbeat session, which gets recycled. The work never came back.

**Rule locked:** Any work that needs a real deliverable from a specialist agent (Atlas, Thrawn, Lando etc.) must use an isolated one-shot cron — not a sessions_send to the heartbeat session. This is now logged as L-022.

## Training Content Angles (for Aria — AI Course Ideas)

These are lessons from today's work that would make great training modules:

**TC-096** — *AI image generation: when to automate and when to keep humans in the loop*
We built an end-to-end image pipeline, tested it, and Ken chose the human-made version anyway. That's a real lesson: AI tools augment creative decisions, they don't replace them. Great for explaining the human-AI collaboration model.

**TC-097** — *Token efficiency as a design principle: why your AI bill is an architecture problem*
The Warden cron was burning 17k tokens per run for pure automation (no reasoning needed). The fix wasn't just a tweak — it revealed 6 root causes that apply to any AI platform. This would be a strong module for "building cost-aware AI systems."

**TC-098** — *The isolated agent pattern: why delegate-and-forget breaks in multi-agent systems*
The Atlas sessions_send failure shows a real trap in multi-agent orchestration: sending work to a session that gets recycled. The fix (isolated cron, one-shot execution) is an important pattern for anyone building with multiple AI agents.

**TC-099** — *Strategy to backlog: the gap that kills AI platform roadmaps*
We found a live example: MinIO was in Atlas's roadmap but had no ticket. The process gap between "strategy doc" and "actual work item" is a common failure mode in any project, but especially in AI ops where things move fast. TKT-0125 formalises the ceremony.

**TC-100** — *The invisible file problem: how to make AI-generated outputs accessible to humans*
Ken couldn't access platform-generated files from his Windows machine. The solution (Drive → MinIO progression) illustrates a real architectural decision teams face: what does your file access layer look like when your AI runs headless on a server?
