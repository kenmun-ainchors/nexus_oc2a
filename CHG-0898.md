## Problem

Telegram DM routing to agent main is fixed (TKT-1003 closed). HEARTBEAT.md
documents that platform/cron/infra/business alerts route to both Ken
(8574109706) and Angie (8141152780) per CHG-0799, but enforcement was
incomplete: `sovereign-alert.sh` (the primary cron alert wrapper) defaulted
to Ken only, and several ad-hoc `telegram-alert.sh` call sites either
hardcoded the single Ken chat ID or used an outdated positional API.

Ken requested: "every platform alert/notification sent via Telegram also
go to Angie, not just Ken." (TKT-1004)

## Changes

1. **`scripts/sovereign-alert.sh`** (highest-leverage fix)
   - Added `DEFAULT_RECIPIENTS="8574109706,8141152780"`.
   - Added `--single-recipient` opt-out flag for tests / one-off pings.
   - Default invocation now passes `--recipients "$DEFAULT_RECIPIENTS"` to `telegram-alert.sh`.
   - Log line now includes recipient list for traceability.

2. **`scripts/health-check.sh`** (MinIO CHECK 18)
   - Fixed the broken positional API call (the script's `$TELEGRAM_CHAT_ID` was never defined).
   - Now uses `--message` + `--recipients "8574109706,8141152780" --silent`.

3. **`scripts/state-health-assert.sh`** (EOD block)
   - Added `--recipients "8574109706,8141152780"` to the `send_telegram_alert` function.

4. **`scripts/skill-audit-cron.sh`** (Shield BLOCK)
   - Added `--recipients "8574109706,8141152780"` to the alert call.

5. **`scripts/tqp-sentry.py`** (TQP violation)
   - Migrated from broken positional API (would have failed with "Unknown arg: TQP_VIOLATION") to the new `--message` + `--recipients` flag API.
   - Routes to both Ken + Angie.

6. **`scripts/telegram-alert.sh`** (header documentation)
   - Added a TKT-1004 note to the header making it explicit that platform/cron/infra alerts MUST use `--recipients "8574109706,8141152780"`.

## Why

- TKT-1004 explicit ask: "every platform alert/notification sent via Telegram also go to Angie".
- Fixing `sovereign-alert.sh` is the single highest-leverage change because every
  cron that uses `sovereign-alert.sh` (11 crons per `openclaw cron list` audit)
  automatically inherits the dual-recipient routing.
- Ad-hoc call sites in `health-check.sh`, `state-health-assert.sh`,
  `skill-audit-cron.sh`, and `tqp-sentry.py` were either silently broken
  (MinIO alert, TQP sentry — both used an old positional API) or routed to
  Ken only.
- `cross-agent-alert.sh` was already correct (CHG-0799 baseline) — left alone.
