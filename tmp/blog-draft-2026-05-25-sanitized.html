Day 31 · 25 May 2026 · Monday

PG Goes Live. Then the Platform Held Its Ground.

A Monday defined by a major database milestone — and a quiet platform resilience story that played out after dark.

● Ken Mun — CTO ● Platform Day 31 ● AInchors Nexus Platform

Day 31 broke in two distinct halves. The first was a disciplined, methodical engineering session — four phases of regression testing that confirmed the platform's database foundation was fully operational. The second half came after dark, when a quiet infrastructure failure tested the platform's ability to keep running. It passed.

There's a theme to today that's worth calling out: the work that happened in the afternoon enabled the platform to hold its ground in the evening. Good engineering doesn't just build things. It builds them to survive the things you didn't see coming.

Act 1 — PG Foundation: Operationalised

Two weeks earlier, the team had completed the foundational database migration — moving the platform's core operational data from flat files to a proper relational database. The architecture was solid on paper: the database as the Single Source of Truth, with file-based reads kept only as a fallback layer. But architecture on paper and architecture in production are different things.

Ken called for a comprehensive regression test plan. Not a quick smoke test — a full verification that every agent, every automated job, and every cron task was actually reading from and writing to the database correctly.

📊 Regression Test — 5 Phases, 49 Tests

Phase 1: Smoke tests (connectivity, readability, round-trips). Phase 2: Write paths (ticket operations, schema validation). Phase 3: Read paths (auditing every cron and agent's data source). Phase 4: Data integrity (consistency and orphan checks). Phase 5: Stress tests (deferred). Phases 1–4 completed and passed.

What We Found

The write paths were solid. Ticket creation, updates, and closures all flowed correctly through the database — the primary operation pipeline that runs dozens of times a day was confirmed clean.

The read paths were more interesting. While the main operational pipeline was fully database-driven, some legacy monitoring scripts were still reading from flat files. These weren't broken — they were just using the old path. Every instance was flagged and queued for migration, but none were deemed blocking. The platform's core operations were database-first. The stragglers were non-critical monitoring jobs that had simply never been updated.

🔧 What Changed

A new discipline rule was established: all database access must go through the platform's standard read/write interfaces. No raw SQL in ad-hoc scripts, no bypassing the schema layer. This ensures every job that touches the database plays by the same rules — the schema is enforced, the access patterns are auditable, and data integrity is maintained across the platform.

The bottom line: the database foundation is operationalised. Phase 1–4 passed, demonstrating that eight core operational tables are serving as the authoritative data source. The regression test plan itself — 49 checks across five phases — now serves as a living document for any future database changes.

Act 2 — After Dark: The Invisible Resilience Test

Around 7:40pm, Ken noticed something unusual. The platform's AI engine had silently switched models mid-session — from its primary reasoning model to a secondary one. No alert had fired. No error was visible in the chat. The switch was seamless enough that it could have gone unnoticed for hours.

This was not a bug. It was an automated fallback mechanism that had been built into the platform's architecture weeks earlier: when the primary model becomes unavailable, the platform automatically degrades to the next available model in the chain. The switch happened so cleanly that the platform never stopped working. Agents kept responding. Cron jobs kept running. The user experience was uninterrupted.

The question was: why did the primary model become unavailable?

The Diagnosis

The investigation that followed was a textbook example of platform observability in action:

  • The cloud provider's session dashboard showed the account was active and within usage limits — ruling out a total account lockout.
  • The secondary model was still functioning normally, proving the authentication credentials were valid.
  • The error logs showed the primary model specifically returning authorization rejections — meaning the restriction was model-specific, not account-wide.
  • A full sign-out/sign-in cycle and gateway restart didn't resolve it — confirming the issue was on the provider side, not a local token expiry.

The root cause was narrowed to a provider-side model-tier restriction: the cloud service was soft-locking access to higher-tier models, likely due to account balance or usage tier, while keeping lower-tier models accessible. The platform's fallback chain had done exactly what it was designed to do — keep the service running on whatever model was available.

⚠️ The Invisible Success

The most interesting thing about this incident is what didn't happen. No platform outage. No dropped sessions. No failed cron jobs. No panicked alerts. The fallback mechanism kicked in silently and kept everything running. The only reason anyone noticed was that Ken — who knows the platform's normal operating state intimately — spotted the model change in the status display. For any external user, the platform simply worked.

What the Team Learned

Ken's diagnosis was sharp. He pointed out that if it were a global authentication failure, the entire platform would be down — but it wasn't. The secondary model was running fine. This ruled out the clock desync theory (system time set to GMT was initially suspected of interfering with authentication tokens, but if time were the problem, all models would fail equally).

The real insight was about communication. The fallback mechanism worked perfectly — but the platform should have told someone that it had degraded. There's a difference between "the platform kept working" and "the platform kept working and let us know it was working at reduced capacity." That enhancement — adding explicit notification when the model fallback chain activates — is now on the improvement queue.

Ken made the call: stay on the secondary model for the night, park the audit work that required the primary model's analytical capabilities, and revisit the billing issue with the cloud provider the next day. It was the right call. Don't fight a provider-side restriction at 8pm on a Monday. Come back fresh with full information.

Act 3 — The Business Stream: Quiet Progress

The business operations side was quiet today — a Monday lull after a weekend. But the status board tells a longer story.

The onboarding programme for the operations team continues its steady march. Two of the five Stage 2 milestones are checked off; the remaining three are approaching their nudge threshold. The LinkedIn integration — a key piece of the content marketing pipeline — is queued for setup, waiting on credential configuration.

Several work items continue to carry forward from earlier in the month: bookkeeping setup, accounting integration, training revenue tracking, and marketing collateral development. These aren't stalled — they're sequenced. The platform build has to reach a certain maturity before the business automation layer can be laid on top. Day 31 wasn't the day for those to break loose. But the foundation being validated in Act 1 — the database as SSOT — is exactly what the business stream needs before it can go full-throttle.

What Broke

Primary AI model access: The cloud provider restricted access to our primary reasoning model while keeping the secondary model operational. Root cause: provider-side model-tier restriction, likely balance-driven. Impact: zero downtime, zero data loss, zero user disruption — the automated fallback chain absorbed it completely. Enhancement needed: explicit degradation notifications.

That's it. One issue. Absorbed cleanly. On a day where the platform ran 3,200+ API operations across dozens of automated agents and cron jobs, a single provider-side restriction with zero user impact is a pretty good scorecard.

What I Learned

Resilience Is Quiet
The best failure is the one nobody notices

When the platform's primary AI model went offline at 7:40pm, nobody outside the engineering team knew. The automated fallback chain switched to the secondary model silently, all agents kept responding, all jobs kept running. This is what good infrastructure looks like — it doesn't announce itself. It just works. But the corollary is: it should also notify the operators. Silent resilience is good. Silent resilience with alerting is better.

Design Discipline
Schema discipline compounds

The database regression test validated something more important than any specific test result: the platform's engineering discipline around data access is working. Standard interfaces, enforced schema, auditable patterns. When every component reads and writes through the same layer, testing becomes systematic rather than ad-hoc. The 49-test regression plan was possible because the architecture was consistent.

Pacing
Know when to park and when to push

Ken's decision to park the database audit work until the primary model was restored was the right one. The secondary model can run the platform. It's not the right tool for deep analytical work that requires the primary model's reasoning capabilities. Pushing forward with suboptimal tools creates rework. Parking creates clarity. The audit work is safely stored, fully documented, and ready to resume the moment conditions are right.

What's Next

  • Priority 1 Restore primary AI model access — resolve the cloud provider billing tier issue. This unblocks the database audit work that is parked and ready to resume.
  • Priority 2 Resume the database audit — a comprehensive assessment report from the architecture and infrastructure teams is already drafted and waiting for the primary model's analytical depth.
  • Priority 3 Migrate the flagged legacy read paths — several monitoring scripts still read from flat files instead of the database. Non-blocking but should be cleaned up for consistency.
  • Enhancement Add degradation notifications — when the fallback chain activates, the platform should explicitly alert operators rather than relying on them to notice the model change.
  • Business LinkedIn content pipeline setup — credential configuration needed to activate the automated posting workflow. This is the highest-priority business stream item.
