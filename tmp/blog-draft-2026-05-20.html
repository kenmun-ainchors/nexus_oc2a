Parity and Recovery: The Silent Bot Crisis
Ken Mun (CTO)

I came in expecting a quiet evening of verification, but found out the platform had been playing a game of silence since dawn. It was one of those days where the "automation" felt like a liability until the right switch was flipped.

The Silence: 06:58 AEST

The day started with a failure I didn't see until the evening. At 06:58 AEST, the Telegram bots went dark. Two auto-restart cycles had already failed, likely triggered by a network flap during one of the Ollama outage windows. For the better part of the day, the platform was essentially a ghost—running, but unreachable via my primary mobile interface.

When I finally triggered the Yoda keyword on Telegram in the evening, it was clear the gateway needed a hard reset. A quick restart brought both bots back online, but it highlighted a fragility in the auto-recovery logic: if the failure happens at the network/gateway boundary, the bots can't "self-heal" because they can't talk to the brain.

⚠️ Incident: Bot Outage Telegram bots offline from 06:58 AEST until evening. Cause: Network flap + failed auto-restart cycles. Resolution: Gateway restart.

The Gap: Telegram vs. Webchat

Once back online, I focused on a fundamental requirement: parity. Telegram shouldn't be a "lite" version of the experience; it should have the exact same rights, routing, and tool access as the webchat. I asked Yoda to audit the differences.

The audit revealed a specific gap in the system configuration. While the tools and model access were identical, the subagent spawn allowlist was missing for the Telegram channel. This meant that while I could talk to Yoda, I couldn't spawn specialized agents like Spark via Telegram—they were defaulting to "same agent only."

// Fix: Set allowAgents to ["*"] for full parity
"subagents": {
  "allowAgents": ["*"]
}

I pushed the change, restarted the gateway, and verified that the spawn capabilities were now aligned. That's not nothing—having the full agent roster available on my phone changes how I can manage the system during "away" hours.

📊 Decision: Full Parity Standardized subagents.allowAgents across all channels. Result: Telegram now has identical agent-spawning capabilities as webchat.

What I Learned

The Danger of Silent Failures: If the monitoring system is tied to the same gateway as the service, a gateway failure creates a blind spot. We need a more robust external heartbeat that alerts me independently of the bots.

Implicit Defaults are Risks: The allowAgents gap existed because the system had a "safe" default. In a high-velocity build, "safe" defaults often become "invisible" bottlenecks. Explicit configuration is the only way to ensure parity.
