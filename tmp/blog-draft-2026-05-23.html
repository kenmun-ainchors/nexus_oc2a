Day 29 — 2026-05-23

The PostgreSQL SSOT Migration

Closing the loop on infrastructure and moving to a single source of truth.
Ken Mun (CTO)
I came into the evening expecting a routine wrap-up of the database migration. Instead, I found a design flaw in our listener scripts that would have hammered the server with 1.2 million useless queries a day. That's not nothing.

The Resource Leak: Polling vs. Push

Forge flagged a note: our listener script was using a basic poll loop. For those not in the weeds, that means every agent was waking up every second to ask the database "Anything new?" If we have 14 agents running this, we're looking at 14 queries per second. That's a waste of CPU and a recipe for latent performance as we scale.

⚠️ PERFORMANCE SIGNAL 1.2 million queries per day just to check for updates. The a-priori assumption was "it works," but "working" isn't "efficient."

I pushed for an immediate fix. No "P2" or "Wait for OC2." We don't build waste into the foundation. Forge pivoted and implemented a proper async LISTEN/NOTIFY pattern. The new pg-listen.sh uses a blocking pattern that waits for the server to push the notification. Result: zero polling queries, sub-second latency.

The Big Cutover: Dual-Write Discipline

With the listeners optimized, it was time for the high-risk move: making PostgreSQL the Single Source of Truth (SSOT). I didn't want a "big bang" migration where we flip a switch and pray. We used a 6-atom cutover plan with strict OWL discipline.

The core of the strategy was dual-writing. I updated db-write.sh to write to Postgres primary while keeping the JSONL files as a fallback. Similarly, db-read.sh was updated to a 3-tier fallback system: Postgres → state_v → JSON file.

# The New Read Hierarchy
1. Attempt PostgreSQL query
2. If fail, attempt state_v fallback
3. If fail, read raw JSONL file

We ran parity checks and a rollback drill to ensure that if Postgres went dark, the system wouldn't blink. The transition was clean. The platform now has a real database backbone without sacrificing the resilience of local files.

Operationalizing the Backlog

Once the migration report was reviewed and approved, I had Forge turn the "Next Steps" into actual tickets. We didn't just leave them as notes in a document; they are now sequenced in the AKB Backlog (TKT-0268 through TKT-0274).

The strategy here is a gradual cutover. The infrastructure is live, and the ticket system is on Postgres, but the other 12 agents are still on JSON. We'll soak test the current setup in Sprint 4—prioritizing stability and NAS backups—before moving the rest of the agents in Sprint 5. Stability over speed.

Closing the Day 28 Loop

A quick housekeeping item: Day 28 had a silent blog failure. I had the reconstructed journal and blog synced back to GDrive. Recovery complete, incident closed. It's a reminder that the system is only as good as its last backup.

What I Learned

1. Don't accept "it works" as a definition of "done." The polling loop worked, but it was expensive. High-performance systems are built by questioning the efficiency of the "working" solution.

2. Gradual cutover beats the Big Bang. Moving the core infrastructure first, then migrating agents one by one, removes the risk of a catastrophic failure. It's slower, but it's safer.
