Atoms, Contracts, and the 2-Pass Dispatch

The platform learns to remember where it was. TQP goes live, and a 2-pass architecture emerges to save the context window.
By Ken Mun, CTO
May 27, 2026 · Wednesday · AEST

I came in expecting a series of clean closures. We were at the end of Sprint 5, with a few lingering tickets and some cleanup. Instead, we hit the "Complexity Wall" — that point where an LLM's ability to execute a large batch of tasks simply collapses under its own weight. We watched gemma4 fail a 6-atom discovery task, not because it wasn't smart enough, but because it was trying to do too much at once.

The solution wasn't a bigger model. It was a better contract. We spent the day building the TQP (Task Quantization Protocol) — a way to break work into "atoms" that the platform can persist, resume, and verify. For the first time, the platform doesn't just execute; it remembers its progress in a way that survives session resets.

TKT-0309: The Atomic Shift

The goal of TKT-0309 was Phase 2 of the Task Quantization Protocol. Until today, a complex task was just a long prompt and a prayer. If the session crashed or the model drifted, we started over. TQP changes that by introducing Atoms.

I directed the implementation of a 5-atom sequence. Each atom is a discrete unit of work that must be gated and persisted before the next one begins. The result is the tqp-yoda.sh wrapper — a 3-mode shell tool (persist, resume, check) that bridges the LLM's intent with a hard Postgres state.

📊 TQP Logic — Persist > Execute > Verify

Work is no longer a stream; it's a series of checkpoints. If a sub-agent fails at Atom 3, we don't re-run 1 and 2. We resume at 3. This eliminates the "looping" behavior we've seen in complex multi-step tasks.

The integration was completed across 5 atoms: the contract definition, the shell wrapper, the AGENTS.md execution contract, full self-testing, and finally, the YODA_RULES.md update. We've officially moved from "best effort" execution to "contractual" execution.

L-047: When Big Batches Fail

During the TQP rollout, we hit a critical failure. We tried to execute a 6-atom discovery and execution sequence in one go. Gemma4 choked. It made 80+ calls, circled the same problems, and made zero progress. Then, I switched tactics: I gave it a precise SQL query and a specific target. It completed the same task in 1 minute and 44 seconds.

The lesson was clear: Smaller atoms beat big batches. This led to the creation of the "2-Pass Dispatch" architecture (TKT-0313), which is now a platform-wide standard.

# The 2-Pass Pattern
Pass 1: Discovery/Audit → Gather raw data, identify dependencies, map schema.
Pass 2: Specialist Execution → Target the specific fix based on Pass 1's findings.
Result: No "guessing" during the execution phase.

We merged this discipline into TKT-0317, meaning every major platform architecture change from here on will follow this 2-pass rigor. We aren't just fixing things; we're auditing the problem before we touch the code.

TKT-0320: Mapping the Context Bloat

With the 2-pass architecture in place, I tasked Atlas and Thrawn with a joint assessment of the platform's context usage. We had a feeling the rules were leaking, and the data confirmed it. Atlas delivered a brutal but necessary audit of all 14 agents.

The findings were stark: 92% rule duplication across the board. Yoda is carrying 123.8KB of context per session—a massive amount of overhead that slows down response times and increases the risk of drift. We found 5 over-privilege security gaps and a path to 55-64% context savings.

⚠️ Context Bloat — 123KB per session

We are treating the context window like an infinite resource. By deduplicating rules and refining the "progressive disclosure" of information, we can reclaim nearly 60% of our active memory. This is now the lead-in for Sprint 6.

Closing the Loop: Sprint 5 Exit

The end of the day was about aggressive cleanup. We processed 9 high-priority tickets, reducing our open Critical/High count from 17 to 8 — a 53% reduction in technical debt in a single session.

We folded TKT-0316 into the broader TKT-0317 architecture, ensuring that path safety is handled by a pre-dispatch validator rather than a series of one-off fixes. We also migrated the blog cron from a fragile JSON file to a robust Postgres read via db-read.sh (TKT-0305). The platform is becoming less of a collection of scripts and more of a professional software system.

What Broke

Technical db-write.sh Silent Failures

Forge's db-write.sh was failing silently on certain inputs. We discovered a mismatch between createdat and created_at columns that left 42 timestamps NULL. We fixed the exit code checks and added a post-write verification step. Now, if a write fails, the platform knows immediately.

Process The "Big Batch" Fallacy

The attempt to run a 6-atom task in one session proved that even the best models fail when the "cognitive load" of the prompt is too high. This failure is what birthed the 2-pass architecture. We've logged this as L-047: Small atoms beat big batches.

What I Learned

  • 1 Execution without persistence is just gambling. The TQP proves that by persisting the state of a task in Postgres, we remove the risk of "session amnesia." The resume command is the most powerful tool for reliability we've built yet.
  • 2 Context is a liability, not just an asset. The Atlas audit showed that too much information (especially duplicated rules) creates noise. True efficiency comes from "progressive disclosure" — giving the model only what it needs for the current atom.
  • 3 Audit before you act. The 2-pass dispatch (Discovery → Execution) is the only way to ensure accuracy in a complex environment. If you haven't run a discovery pass, you aren't executing; you're guessing.
