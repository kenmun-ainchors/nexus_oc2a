Day 28 · 21 May 2026

The Autonomous Engine

Six tickets, a database, and a governance pattern — with the CTO mostly watching from Telegram.
By Ken Mun, CTO · AInchors

I came into Thursday expecting a quiet day. The Claude credits are still depleted — I'm running the entire platform on free-tier Ollama models. My main involvement would be a few Telegram check-ins: approve a LinkedIn post, clear a couple of RTB items, maybe check the stand-up.

What I got was the most autonomous day the platform has ever had. Six tickets closed. PostgreSQL 16 deployed onto the Mac Mini. The State Checking Pattern codified into RULES.md as a non-negotiable. A document generation pipeline delivered. And Angie — silent for seven days — sent two messages to Aria.

I spent maybe 20 minutes actively involved. The agents did the rest.

📊 Day 28 by the numbers 428 turns. $0.00 API spend. 6 tickets completed. 1 database deployed. 4 tilde-path write failures (same bug, nth occurrence). 102 false-positive monitoring alerts from known drift — real signal buried in noise. And the return of Angie after a 7-day silence.

Act 1: "Morning Sprint: TKT-0232 and the Write Bug That Won't Die"

The day started at 05:09 AEST with three Telegram messages from me — a quick RTB sprint. TKT-0232 needed fixing: the Notion sync script was missing, creating drift between local ticket state and Holocron. The agent found the gap, wrote the script, and reported it resolved within 40 minutes.

Except the write failed.

scripts/notion-sync-fix.sh — 6,802 characters of perfectly good shell script — vanished into the void because the agent used a tilde path instead of an absolute one. CHG-0281 has been a NON-NEGOTIABLE rule in RULES.md for weeks: always use full filesystem paths, never tilde shortcuts. Isolated sub-agent sessions don't expand tildes. The rule says never use them. The agents keep using them.

⚠️ The tilde-path bug is now a platform reliability issue Four write failures today across three different agents. CHG-0281 is documented, communicated, and ignored by sub-agents in isolated sessions. A rule that isn't enforced isn't a rule — it's a suggestion. This needs code-level enforcement: the write tool itself should reject tilde paths, or the gateway should rewrite them at spawn time.

The RTB items were functionally resolved — the content was produced, just not persisted to disk. But "functionally resolved" isn't the same as "done." When your platform's most frequent bug is path syntax, you have a tooling problem, not a discipline problem.

Act 2: "Ten CHGs, One Glaring Alarm"

At 08:05 the morning stand-up landed in my webchat. Ten CHGs since the last stand-up — backup fixes, model audit, gateway restart reliability, async rules, LinkedIn SSOT, Drive discipline, blog governance fix, journal boundary fix. Heavy infrastructure week. No breaking changes. Strong cadence.

Then the alarm: 102 cron_run_fail events flooding the monitoring dashboard. All from 12 intentional interim Ollama drift items — CHG-0363 model assignments that were approved but still flagged as errors by the observability layer. The monitoring system was crying wolf 102 times a day, and any real incident would be buried in the noise.

The stand-up agent asked for a decision: file a CHG to add a baseline exception for those 12 known model assignments, silencing the false positives without touching real monitoring. I haven't made that call yet. It's sitting in my queue for Friday.

🚨 Alert fatigue is a real risk 102 false positives per day means the monitoring dashboard is effectively useless for detecting actual incidents. If a real crash fires, nobody will notice it in the noise. This needs resolution before the weekend.

Act 3: "LinkedIn, End-to-End, While I Was on Telegram"

Mid-morning, I sent five Telegram messages — discovery, drafting, approval — for a new LinkedIn post. The pipeline ran end-to-end without me touching webchat:

  1. Topic discovery — found a new angle for the AIOps content series
  2. Drafting — 1,737 characters of first-draft copy
  3. Governance gate — Shield, Lex, and Sage all cleared (Shield clear, Lex conditional, Sage conditional)
  4. Queue registration — CONTENT-LI-20260521 added to the LinkedIn queue with my approval timestamp
  5. Two more tilde-path write failures — draft file and queue JSON both failed to persist (same bug)

The content survived in-memory and made it into the queue. The governance gate worked. But the pipeline has a persistent gap: content that clears governance but can't be written to disk isn't really "published." It's approved and stranded.

That's it. Two short messages. But they ended a seven-day silence — the longest gap since the platform launched. Angie's last contact was May 14, when she asked Aria to schedule a meeting with me. I don't know what she said yet — Aria's responses live in the business stream, and the technical gateway logs don't capture agent-to-agent content. But the re-engagement itself matters. The business half of AInchors may be about to wake up.

Act 5: "PostgreSQL Lands on the Mac Mini"

The afternoon was a quiet infrastructure sprint — three significant deliveries in 90 minutes, none of them requiring my involvement:

TKT-0196 — Three Work Types Rule + Work Currency Routing Table (Phase 1). This codifies how work gets classified and routed across the agent team. Three work types, each with its own routing rules. Phase 1 is the classification system; Phase 2 will be automated routing based on the table.

PostgreSQL 16 + 5-tier schema. This is the big one. PostgreSQL is now running on the Mac Mini M4 with a 5-tier schema deployed. It's the platform's first structured relational database — the foundation for P1 semantic memory (T4), the migration off flat JSON files, and eventually the Holocron backend itself. The deployment was clean. What's missing: documentation of what lives in each tier, and the migration tooling to move data from JSON to Postgres.

Sources of Truth Register. A canonical registry of every SSOT data source in the platform. Combined with the State Checking Pattern (coming in Act 6), this gives every agent both the WHAT — where truth lives — and the HOW — how to verify before acting.

📊 Infrastructure milestones PostgreSQL on bare metal is a different thing from PostgreSQL in a Docker container. This is step one of the data layer migration that the Technology Strategy & Roadmap calls for in P1. The 5-tier schema needs documentation, but the fact that it deployed cleanly on a Thursday afternoon with zero drama is a good signal.

Act 6: "Forge Can't Find the Files"

Not everything landed cleanly. TKT-0198 — the JSON-to-Postgres migration — failed because Forge couldn't find the state files. It was looking in workspace/forge/, which was empty after cleanup. The files live in workspace root.

This is the workspace isolation problem, now confirmed across multiple agents. Atlas got the fix earlier this week (TKT-0235) and writes correctly to workspace root. Forge hasn't received the same treatment. When a sub-agent spawns in an isolated directory, it can't see the files it needs to modify — and the result is a 31-minute wait for a failure report.

The pattern is clear: every agent that needs to write to shared state needs the workspace discipline fix. It's not a one-and-done — it's a per-agent configuration task that hasn't been systematically applied.

Act 7: "Check Before You Act — Now It's a Rule"

The day's final delivery was TKT-0182: the State Checking Pattern. It formalises a cycle that should have been implicit: READ → VALIDATE → EXECUTE → VERIFY.

Atlas wrote the canonical reference document to workspace root — confirming that TKT-0235 (workspace discipline) is working for Atlas. The pattern was then encoded into RULES.md as a non-negotiable section: violation of the State Checking Pattern is now a Definition of Done failure.

The irony isn't lost on me. The same day we codified "check before you act" into the rulebook, Forge failed because it didn't check where the files were before trying to migrate them. The pattern is correct — it just arrived 30 minutes too late to save TKT-0198.

📊 State Checking Pattern Four steps, now mandatory for all stateful operations: (1) READ current state, (2) VALIDATE the read succeeded and data is current, (3) EXECUTE the change, (4) VERIFY the change persisted. Violation = DoD failure. Combined with the Sources of Truth Register, this closes a major governance gap.

What I Learned

1. The tilde-path bug has graduated from "annoying" to "platform reliability issue." Four failures in one day, across three agents, on a rule that's been NON-NEGOTIABLE for weeks. Rules without enforcement are suggestions. The write tool needs to reject tilde paths at the code level. A regex check before every filesystem write would solve this permanently.

2. The platform can execute autonomously now. Six ticket completions. PostgreSQL deployment. LinkedIn pipeline end-to-end. State Checking Pattern codified. Sources of Truth Register created. I spent ~20 minutes actively involved all day. The rest was agents picking up work from the queue and delivering without hand-holding. That's the vision — it's starting to work.

3. Infrastructure without migration tooling is infrastructure that isn't being used. PostgreSQL is deployed. The schema is live. But TKT-0198 is blocked because Forge can't find the JSON files to migrate. The gap between "deployed" and "operational" is tooling. Every infrastructure deliverable needs a companion migration task, not just a deployment task.

4. Angie's silence broke at 13:56. I don't know what she said yet. But seven days of radio silence ending with two Telegram messages is a signal. The business stream may be about to restart. I need Aria's brief to know what's happening on that side of the house.
