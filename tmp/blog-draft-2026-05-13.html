The Key That Died Overnight

An API key was quietly invalidated while we slept. Six cron jobs went dark. We recovered in under three minutes - then spent the rest of the day making sure the platform grows up a little more.

By Ken Mun, CTO — AInchors

I came in at 12:39 to a system health check, expecting the usual clean status. What I got instead was six concurrent cron failures lighting up across completely different jobs - the relay poller, the journal writer, Warden compliance, Forge validation, the allowlist sync, and the morning stand-up. Six independent jobs, all failing at once. That's not a coincidence; that's a shared root cause.

The answer was simple, once I found it: our upstream API key had been invalidated overnight. Not a local deletion, not a local misconfiguration - a server-side event. The key was still sitting in the auth file exactly where it always was. But every call to the model provider came back 401 Unauthorized. The platform had been running on fumes since the morning gateway restart exposed the dead key. After that, the incident was already over. The rest of Day 19 was about maturity work - tightening content rules, auditing the knowledge base, seeding the next sprint, and locking a governance decision that had been quietly needed for weeks.

Act 1: The Dead Key — From Six Failures to Zero in Three Minutes

The root cause analysis was straightforward once I had the data. The gateway auto-restarted at 07:09 AEST - triggered by a blog cron stall from earlier that morning. The restart forced a fresh auth check, and the fresh auth check immediately exposed what cached sessions had been hiding: the API key had been invalidated upstream at some point overnight.

Before the restart, running sessions were using cached authentication tokens. They were working fine. The moment the gateway came back up clean, every agent that tried to make an Anthropic call got a 401. All six cron jobs failing simultaneously wasn't six separate problems - it was one problem showing up in six places at once.

🚨 Root Cause — INC-20260513-KEY

The API key was invalidated upstream overnight. Local auth file unchanged. Gateway restart at 07:09 exposed the dead key by forcing fresh auth checks. All six morning crons failed as a result. MTTR: under 3 minutes from diagnosis to re-triggered.

The fix was to provision a new API key and run openclaw models auth to register it. I re-triggered the five failed crons manually - Warden, Journal Incremental Writer, Forge Fallback Chain Validation, Allowlist Sync, and the Relay Queue Poller. The Morning Stand-Up I deliberately skipped; running it at 12:40 would have generated stale timing that made no operational sense.

The one thing I flagged for follow-up: the infrastructure agent had cloud-model calls in some of its crons but an Ollama-only auth profile. That's a latent risk. If the gateway restarts again before that gets patched, those jobs will fail for a different reason than today's. It goes on the Sprint 4 list.

Act 2: Content Rules Locked — No More Leaking the Architecture

The LinkedIn draft for Week 2, Post 3 came back rejected. The topic was right - multi-agent trust and how AI systems verify each other's outputs. But the draft was referencing internal system names directly. Our orchestrator by name. Our governance layer by its actual label. The exact architecture of how our platform works.

That's not a content error - that's a security posture error. We're a consulting firm. Our system design is our competitive IP. Publishing the internal wiring of our platform in a LinkedIn post is the kind of thing that looks harmless until it doesn't.

Two hard rules locked for the content agent effective immediately:

  1. No em dashes. Use a hyphen when punctuation is needed, or restructure the sentence.
  2. Never reference internal product or agent names in public content. Use generic labels: "our orchestrator", "the governance layer", "our coordination framework".

The revised draft kept the same core argument - that multi-agent trust requires explicit verification, not assumption - but expressed it without exposing anything proprietary. That version was approved and wired into Thursday's 07:30 AEST posting cron.

Act 3: The Cloudflare Tunnel — Built, Tested, and Parked

The access architecture document from Monday identified Cloudflare Tunnel as the right answer to a persistent friction point: I can't push files directly to the workspace from my Windows machine. Every time I need to drop an image or an asset, it has to come through Telegram - which works, but isn't the right long-term pattern.

Today we built it. The tunnel came up, DNS routes were configured for the three target endpoints, and a LaunchAgent was installed to keep it running. Technically complete.

Then I stopped it.

⚠️ Security Gate — TKT-0155 Parked

The Cloudflare tunnel was fully operational but had no access control layer. Cloudflare Access (email and OTP authentication) had not been configured - meaning the three URLs were publicly reachable. The tunnel was stopped immediately. It stays offline until the DNS namespace settles and CF Access is configured properly. Reminder set for Wed 20 May 08:00 AEST.

The Cloudflare DNS namespace for the domain isn't fully propagated yet. Building the access control layer before the DNS is stable means building on shaky ground. There's no urgency forcing us to accept that risk. The Telegram workaround exists. We'll pick this back up next week with the full security stack in place from day one.

Act 5: The Two-Yoda Problem - And How We're Fixing It

Late in the session, a real architectural issue surfaced. I noticed that the sandbox runbook produced during an earlier Telegram interaction was a bare scaffold, while the one produced tonight in the main webchat session was comprehensive. Same agent, same task, dramatically different output.

The reason: they're not the same agent. The Telegram session and the webchat session are isolated contexts. Decisions made in one don't propagate to the other. Context built up in one doesn't exist in the other. When a complex task runs in Telegram, it runs without the full operational history that the main session carries. The outputs reflect that.

"The two-Yoda problem is real and architectural."

The permanent fix - making the orchestrator channel-agnostic so context is unified regardless of where a request comes in - goes on Sprint 4. That's TKT-0160.

The immediate fix is a policy rule, locked tonight. Channel Discipline (CHG-0303):

  • Webchat / main session - Authoritative. All decisions, change records, approvals, and strategy happen here.
  • Telegram - Status checks, quick questions, time-sensitive notifications only. No architecture decisions, no CHG records, no approvals.
📊 CHG-0303 — Channel Discipline

Locked in YODA_RULES.md (v2.1.0) as section R3a. Sprint 3 operational fix complete. Sprint 4 TKT-0160 queued for permanent architectural solution. Until that lands, the single-source-of-truth principle for decisions is enforced by rule.

What I Learned

1. Cached auth hides real failures. The API key was invalidated overnight and we didn't know for hours, because cached sessions were still working. The first signal was the gateway restart exposing the dead key to fresh checks. Cached state is a reliability cushion that can also be a blindfold.

2. Build the security layer before you open the door. The Cloudflare tunnel worked perfectly. We stopped it anyway, because running a tunnel without Cloudflare Access is running a tunnel without locks. The 20 extra minutes to configure access control properly is worth far more than the week of exposure if we'd left it live.

3. Generic language protects IP and still makes the point. The rejected LinkedIn draft and the approved one covered the same argument. One named our internal systems explicitly. The other didn't. The argument didn't weaken - the exposure did.

4. Context isolation produces quality gaps. The two-Yoda problem is a concrete example: isolated sessions produce isolated quality. You can't maintain high output consistency across channels if each channel is running on a different knowledge base. The fix has to be architectural, not just behavioural.
