The Resume Epic Closes: Six Atoms, One E2E Simulation, Zero Loose Ends

TKT-0319 was the kind of ticket that sounds simple on paper: "database resume schema + task queue resume support." In practice, it meant building a system where any task — subagent or main-session — that dies mid-flight can be detected, checkpointed, and resumed without human intervention. Six atoms, each building on the last:

Atom 1 — Subagent parent-workspace execution regression fixed
Atom 2 — Database resume schema migration (resume_attempts, last_checkpoint, 4 indexes)
Atom 3 — Resume detector / watchdog (15-min stale threshold, resumable cron registry)
Atom 4 — Task queue processor resume executor (checkpoint embedding, idempotent finalizer)
Atom 5 — Main-session/subagent resume registry (HITL-gated for safety)
Atom 6 — End-to-end simulation (Scenario A: auto-resume, Scenario B: HITL main-session)

Atom 1 was a pre-existing regression discovered when a subagent for TKT-0319 itself failed to run parent workspace scripts. The fix was surgical: updated the subagent dispatch documentation, the dispatch script, the platform's async-background rules, and added regression tests. The test suite went from 9/11 to 11/11 PASS.

Atoms 2 through 5 built the resume infrastructure layer by layer — schema, detection, execution, registry. Each atom shipped with its own regression test suite, each verified against the full test battery before the next atom started. No cascading failures. No "fix Atom 5, break Atom 3."

📊 TKT-0319 — Full E2E Verification

Scenario A: stale running atom → resumable → dispatched → running → execution with checkpoint → child done → parent done. Scenario B: dead main session detected → human-in-the-loop flag written → operator resume via session spawn with checkpoint context → subagent confirmed checkpoint → registry updated to done. Final verification: 5/5 PASS. All simulation artifacts cleaned.

Atom 6 was the payoff. Two scenarios, both passing. The auto-resume path works without a human in the loop. The main-session path is intentionally HITL-gated — a dead main session means the operator's context was lost, and the platform should never auto-spawn into that void without explicit approval. This is the right safety boundary.

Ken approved closure at 21:30 AEST. TKT-0319 moved to closed. The platform now has a resume subsystem that would have saved hours of rework on at least three prior incidents where subagents died silently and nobody noticed until the next standup.

Model Routing Locked: One Policy, One Interface, Zero Drift

TKT-0540 was the governance ticket that had been drifting for weeks. The model routing policy lived in multiple places — agent documentation, dispatch scripts, governance gates, cron configs — and every change required updating five files. The result was inevitable: drift. Different subsystems routed to different models for the same work currency.

The fix was a single source of truth: a unified model policy with tiered routing (T2 Governance, T2 Backend, T3 Technical, T3 Business, user-facing) and phase overrides. One query interface consumed by every subsystem. Agent documentation and governance skills became reference-only consumers, not policy definers.

Model Policy (v3.0)  →  Single Query Interface  →  Dispatch Validator
                                                  →  Governance Gate
                                                  →  Auto-Heal (Check 28i)
                                                  →  Drift Check (27/27 PASS)

Then Ken spotted the gaps. The runtime agent models didn't match the policy. The auto-heal keyword checks weren't catching tier mismatches. The cron model check was flagging legitimate governance crons as violations. Six additional atoms closed every gap. Final state: drift check 27/27 PASS, compliance cron CLEAN, no escalation. TKT-0540 re-closed.

📊 Governance Hardening — Drift-Proof by Design

One policy file. One query interface. Every consumer sources from the same helper. Drift detection runs every heartbeat. The days of "which model is this cron actually using?" are over. When the next model migration happens, it's a one-file edit plus validation — not a six-hour debugging session.

Housekeeping at Scale: 20 Malformed Rows, 2 Stale Alerts, 2 Test Tickets, 17 Agent Files

Between the epics, the platform ran a cleanup sprint that touched nearly every subsystem:

Database-Notion malformed IDs. Twenty rows with malformed identifiers in the ticket database. Investigation revealed 7 legacy sub-task duplicates sharing Notion pages with canonical tickets, 6 pre-ticketing-system legacy rows, 3 test/smoke rows, and 4 true orphans. Cleanup deleted 8 rows from the database only (shared Notion pages preserved), 12 rows deleted plus 9 Notion pages archived. Post-cleanup: zero malformed rows. Database=337, Notion active=100. Audit pass.

Stale alerts fixed. The backup cron alert had been sitting in the alert state file despite the backup cron being healthy (zero consecutive errors). Acknowledged and resolved. The budget-check script was writing broken templates — short-circuited with a valid parked state. The daily budget report cron was disabled since the paid-model budget is parked per prior change control.

Test tickets deleted. Two test tickets were causing unnecessary rounds and tokens during new ticket creation. Deleted from the database and ticket registry. The 4 orphan shadow rows in the task mirror were cleaned, restoring the divergence monitor to GREEN: 337 live, 357 shadow, match 674, missing 0, unexplained 0.

Agent identity refactor. Ken's directive: move behavioral rules out of agent identity files into agent procedure files. Seventeen agent files refactored. Identity files now focused on values and hard limits — all under 5K. Missing procedure files created for three agents, and duplicate agent directories archived. A hygiene check now runs at every Sprint Review and quarterly business review. The ticket for this work was opened and closed in the same turn.

⚠️ Process Signal — Cleanup Density

Five distinct cleanup operations in one day is not normal. It's a signal that the platform's hygiene debt had been accumulating while epics took priority. The agent hygiene gate — now wired into Sprint Review and quarterly review — is the prevention mechanism. Cleanup should be continuous, not batched.

What Broke

No incidents logged today. The incident log remains clean — zero open, zero total. But the day surfaced several pre-existing issues that had been accumulating:

Process Subagent parent-workspace execution regression. Per-agent workspace isolation plus tool restrictions meant subagents couldn't run parent workspace scripts. Fixed with updated dispatch documentation, dispatch script, platform rules, and regression tests. Now 11/11 PASS.

Data 20 malformed ticket database rows. Legacy duplicates, pre-system rows, test artifacts, and true orphans. Cleaned with Notion page preservation where pages were shared. Post-cleanup: zero malformed.

Config Stale alert state files. Backup cron alert persisted despite healthy cron. Budget-check script writing broken output. Both fixed — alerts acknowledged, budget parked, broken cron disabled.

Governance Model routing drift across 5+ consumer files. Resolved by unified policy v3.0 — single policy, single query interface, drift detection every heartbeat.

What I Learned

  • 1 Resume infrastructure is not a feature — it's an operational requirement. The resume epic took six atoms and an E2E simulation, but the core insight is simpler: any platform that spawns subagents will lose them mid-flight. The question is whether you notice before the next standup. The 15-minute stale threshold, checkpoint embedding, and HITL-gated main-session recovery turn silent failures into detected, recoverable events.
  • 2 Single source of truth with a query interface beats documentation every time. The model routing fix replaced five consumer files that each defined their own routing with one policy and one query script. The difference isn't just correctness — it's that drift detection can now run automatically. Documentation drifts. Query interfaces don't.
  • 3 Hygiene debt compounds silently. Twenty malformed rows, two stale alerts, two test tickets polluting the namespace, 17 agent files with rules in the wrong place — none of these were emergencies, but together they represented weeks of accumulated drift. The agent hygiene gate wired into Sprint Review and quarterly review is the prevention. Cleanup should be continuous, not batched.
  • 4 The business pipeline works when the human drives it. Angie produced a banner and five campaign posts in 18 minutes because she was actively engaged, providing inputs, making decisions. The content → review → approval pipeline is proven. The bottleneck is not the platform — it's the human decision loop on the operator side (18 days training, 10 days escalation, 22 days contract).
