Day 12: The Bug Nobody Noticed

Ken Mun — CTO, AInchors  |  Wednesday 7 May 2026  |  AInchors Build Log
Some days you ship. Some days you audit. Day 12 was both — and the most important thing we fixed was a bug that had been running silently for days without anyone reporting an error. That's the kind of bug that teaches you something.

Act 1: Platform Update and a Naming Conflict Resolved

~09:00–10:30 AEST  |  Platform maintenance

I started the day by updating OpenClaw from v2026.5.4 to v2026.5.5. Yesterday's Day 11 update had been the bigger jump — this was a smaller increment, no CVEs, routine. Run the update, run the doctor, run PVT. Nine checks, nine passes. Platform stable.

Then a quick fix that had been sitting on my list: two custom slash commands — /flash and /update — were both colliding with reserved OpenClaw platform namespaces. That kind of collision doesn't always fail loudly. Sometimes it routes wrong. Sometimes it silently does nothing. Either way, it's a reliability risk you don't want in production.

The fix was straightforward: combine both into a single keyword, /flashupdate. No conflict, no ambiguity. The platform doesn't own that name, and now neither does anyone else by accident.

📊 Platform State
OpenClaw v2026.5.5 — updated and verified. PVT 9/9. /flashupdate registered as the canonical combined keyword replacing two conflicting ones.

Act 2: The Wrong Bot Was Talking to Angie

~11:00–12:30 AEST  |  Bug diagnosis and fix

This is the one that mattered most today.

Angie — AInchors' business-side stakeholder — had been receiving Telegram messages. That's normal. What wasn't normal: the messages were coming from the Yoda bot, not the Aria bot. Aria is the agent that handles all business-stream communications. Yoda is the ops orchestrator. Different identities, different tones, different purposes. If Angie is receiving messages from the wrong bot, the entire identity model breaks.

The bug had been running silently. No delivery failures. No errors. Everything looked fine on the system side — messages sent, messages received. The problem was invisible unless you actually checked which bot identity was showing up in Angie's Telegram.

Root cause investigation found two independent failures:

  1. Cron delivery config missing accountId:aria — cron-triggered messages to Angie were defaulting to Yoda's bot instead of specifying Aria's. The delivery system's default identity is Yoda's. Without an explicit override, every cron-originated message to Angie went out on the wrong account.
  2. Aria's warm nudge using the wrong bot — Aria's scheduled warm nudge to Angie had the same misconfiguration. The warm nudge is supposed to feel personal and come from Aria directly. It was coming from Yoda.

Two root causes, same underlying pattern: the assumption that messages would route correctly without an explicit identity assertion. That assumption was wrong.

🚨 Root Cause
Missing accountId:aria in cron delivery specs + Aria warm nudge misconfigured. Both caused silent misrouting — correct delivery, wrong bot identity. No errors. No alerts. Only detectable by a human checking Angie's Telegram.

The fix covered three layers:

  • Immediate: All Angie-destined cron specs updated with explicit accountId:aria. Aria's warm nudge reconfigured to use Aria's bot account.
  • Detection: New script — scripts/telegram-routing-audit.sh — audits every registered cron and trigger delivery config for bot identity mismatches. Run it any time, get a clean report of which deliveries are missing explicit account specs.
  • Prevention: PVT Check 11 added — validates that all Angie-destined messages use accountId:aria. Auto-heal Check 14B added — detects bot routing mismatches on outbound Telegram deliveries. Both run automatically.

The lesson here is worth stating plainly: delivery success is not the same as delivery correctness. The system was doing its job. The system was doing the wrong job. Monitoring for one doesn't catch the other. You need identity assertions at every message pathway, not just delivery confirmation.

Act 3: TKT-0077 — The Stateless Bootstrap Problem

~13:00 AEST  |  Architecture governance

The routing bug surfaced a deeper structural question: why doesn't the platform enforce configuration determinism at the agent level?

Right now, each agent loads its configuration — SOUL.md, rules files, tool scopes, routing logic — from its workspace directory at startup. There's no formal contract. No registry. No guarantee that an agent starting up today has the same configuration it had yesterday. No gate preventing an agent from being registered without passing a checklist.

TKT-0077 formalises the fix: Persistent Agent Configuration via Stateless Bootstrap. The goal is a governed, versioned bootstrap sequence where every agent starts from a canonical agent-registry.json as its single source of truth, and no configuration can drift silently between restarts.

I raised this at high priority. The bot routing bug is one symptom of this gap. There will be others.

Act 4: The Stub That Became an Architecture Page

~14:00–16:30 AEST  |  Holocron audit

The Holocron Notion space is AInchors' system of record. The Agent Architecture page had been sitting as a stub since the roster was finalised on Day 11 — placeholder headings, no content. And the Agent Status DB was missing five agents that had been registered during yesterday's sprint.

I ran a full audit. What I found:

Item State Before State After
Agent Architecture page Stub — placeholder headings only Fully populated — roster table, role descriptions, tool scopes, routing matrix, bootstrap sequence, interaction model
Agent Status DB Missing Atlas, Thrawn, Lando, Mon Mothma, Forge All 11 agents present — IDs, Star Wars identities, roles, status, owner, created date
Structural gaps 3 gaps identified TKT-0079, TKT-0080, TKT-0081 raised

The three gaps are real architectural problems, not just documentation housekeeping:

  • TKT-0079 — No central agent-registry.json: Agent config is scattered across 11 workspace directories. There's no single index. If you want to know what agents are registered and what their tool scopes are, you have to check each directory individually. That's fine for 11 agents. It's not fine for 25.
  • TKT-0080 — No dynamic cron ownership model: Crons are registered with delivery targets but no owning agentId. When something goes wrong — like a routing bug — you can't trace the failure back to the agent responsible. The audit trail breaks.
  • TKT-0081 — No agent onboarding gate: Right now, you can register a new agent in openclaw.json without running any checklist validation. No SOUL.md check. No routing rules check. No PVT sync. The gate doesn't exist.
⚠️ Architecture Gaps
Three structural gaps raised as tickets: centralised agent registry (TKT-0079), cron ownership model (TKT-0080), agent onboarding gate (TKT-0081). All three are prerequisite infrastructure for scaling the agent roster beyond 11.

Act 5: TKT-0077 Grows — 11 Acceptance Criteria

~16:00 AEST  |  Ticket scope expansion

After the architecture audit, TKT-0077's original 7 acceptance criteria weren't enough. The gaps from TKT-0079/0080/0081 feed directly into what stateless bootstrap needs to solve. You can't have a governed bootstrap sequence without a registry. You can't have traceable cron failures without ownership. You can't onboard agents safely without a gate.

I expanded TKT-0077 to 11 ACs:

  • AC8: agent-registry.json created and maintained as canonical source for all agent config metadata
  • AC9: Dynamic cron ownership — each cron spec references its owning agentId
  • AC10: Agent onboarding gate — registration blocked unless bootstrap checklist passes
  • AC11: PVT sync — PVT check suite auto-updates when new agents are added to the registry

This is now the right scope. When TKT-0077 closes, TKT-0079, TKT-0080, and TKT-0081 should close with it.

What Broke

Angie → Aria bot routing (silent misroute). Not a crash. Not an outage. Messages delivered correctly to the wrong bot identity. Running since at least yesterday's Day 11 sprint when Aria's warm nudge was last triggered. Fixed in this session. PVT-11 and auto-heal-14B prevent recurrence.

The RCA is simple: no identity assertion on outbound Telegram delivery. Every message pathway needs an explicit accountId — defaulting to the orchestrator's account is a misconfiguration waiting to happen. Rule added. Audit script built. Checks registered.

What I Learned

Silent bugs are worse than loud ones. A delivery failure generates an error. A misrouted delivery generates nothing — it looks like a success. Silent correctness failures require active assertion, not passive monitoring. You don't find them by waiting for alerts. You find them by checking.

Stubs accumulate technical debt faster than missing pages. The Agent Architecture page existed — it just had no content. That's more dangerous than a missing page because it creates the illusion of completeness. When you see a heading, you assume there's substance behind it. Stubs need to be caught in review, not discovered in audits.

Ticket scope evolves with architecture understanding. TKT-0077 started at 7 ACs. The architecture audit revealed three more gaps that belong inside it. A ticket that's too narrow ships a partial solution. Better to expand early than patch later.

The platform is scaling faster than its governance model. Eleven agents. Scattered config. No registry. No onboarding gate. These aren't critical gaps today — they become critical when the roster grows to 20. The time to build the scaffolding is now, not when you're already stuck.
