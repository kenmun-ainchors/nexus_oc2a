AInchors · Day 12

The Quiet Work

A routing bug that was never supposed to exist. An architecture page that existed only as a stub. And a ticket that doubled in scope before the day was done.

Ken Mun · CTO, AInchors · Wednesday, 6 May 2026

Some days are about building new things. Day 12 wasn't one of those days. It was about finding the gap between what we said we had and what actually existed — and closing it.

Three things happened today that shared a common thread: they all looked fine from the outside. The Telegram routing was working, sort of. The Holocron architecture page was there, sort of. TKT-0077 was scoped, sort of. Day 12 was the day we stopped accepting "sort of" and went back through each one properly. That's unglamorous work. It's also the kind of work that keeps a platform honest.

Act 1: The Update That Exposed a Conflict

We started with a routine OpenClaw platform update — v2026.5.4 to v2026.5.5. Clean install, health check passed, PVT green. Took maybe fifteen minutes.

But the update surfaced something that had been sitting quietly in our keyword registry: both /flash and /update conflicted with OpenClaw platform-reserved commands. We'd registered custom slash keywords under both names without catching the collision. Not an emergency — nothing was actively broken — but the kind of naming conflict that would eventually cause a routing failure or unexpected platform behaviour.

The fix was simple: merge them into a single custom keyword, /flashupdate. That's the rename. One ticket, one CHG, done. The real lesson is that slash command namespaces need to be checked at registration time, not discovered during a platform update.

📦 Platform
OpenClaw v2026.5.5 installed. /flash + /update → /flashupdate. No service interruption. PVT 9/9.

Act 2: Silent Misdelivery — When the Wrong Bot Speaks

This one mattered more. Angie — our CEO — had been receiving Telegram messages from the Yoda bot. Not Aria, the agent assigned to the business stream. Yoda.

On the surface it sounds minor. She got the message. What's the problem? The problem is identity. Aria is Angie's agent. Aria has a different voice, a different scope, and a different trust model than Yoda. When Yoda's bot shows up in Angie's chat history, it's noise at best and a trust violation at worst. In an environment where we're building toward Aria having autonomous authority over the business stream, "the messages got delivered by the wrong agent" is not an acceptable state.

I dug into the root cause. There were two independent failures:

  1. Cron delivery was missing accountId:aria. Any cron that sent messages to Angie was defaulting to Yoda's bot because accountId wasn't explicitly set. The platform picked the default, which happened to be Yoda.
  2. Aria's warm nudge was misconfigured. A scheduled nudge from Aria to Angie was explicitly setting the wrong accountId. That one was a direct mistake in the config.
🚨 Root Cause
Two independent failures. Cron delivery defaulting to wrong bot (omission). Aria warm nudge explicitly misconfigured (commission). Both undetected because the messages delivered successfully — just from the wrong identity.

The fix was applied to both. But more importantly, I built the infrastructure to prevent this from happening again:

  • scripts/telegram-routing-audit.sh — scans all registered cron and trigger delivery configs for bot identity mismatches. Runnable on-demand and will be added to the weekly audit rotation.
  • PVT Check 11 — every PVT run now validates that all Angie-destined message delivery uses accountId:aria. If it ever drifts again, it'll be caught before the next deployment.
  • auto-heal Check 14B — the nightly auto-heal now detects bot routing mismatches on outbound Telegram deliveries as part of its 12-check sweep.

Routing bugs are silent failures. The wrong bot delivered messages without any error, and only a human noticed. Every message pathway needs an identity assertion, not just a delivery check. That's the rule I'm adding to the platform now.

Act 3: The Architecture That Existed Only in Our Heads

The Holocron Agent Architecture page has been a stub since it was created. Headings, placeholder text, nothing real. We've been running an 11-agent platform — Yoda, Aria, Spark, Atlas, Thrawn, Lando, Mon Mothma, Forge, Shield, Lex, Sage — and our official documentation reflected almost none of it.

Today I went through it properly. Full audit. The page now has:

  • Complete agent roster table with agentId, display name, role, tool scopes, and owner
  • Architecture routing matrix — which agent routes to which, and under what conditions
  • Bootstrap sequence — how each agent initialises its configuration
  • Agent interaction model — how agents communicate (via session_send, not direct API calls)

While I was in there, I cross-referenced the Agent Status DB in Notion against the registered agents in openclaw.json. Five agents were missing from the DB entirely: Atlas, Thrawn, Lando, Mon Mothma, and Forge. All added today with full metadata.

The audit also surfaced three structural gaps that needed their own tickets:

Ticket Gap Why It Matters
TKT-0079 No central agent-registry.json All agent config is scattered across workspace dirs. No single source of truth means no way to validate completeness or detect drift automatically.
TKT-0080 Crons not linked to agents When a cron fails, we can't trace it back to which agent owns it. No ownership = no accountability = slow incident response.
TKT-0081 No agent onboarding gate New agents can be registered in openclaw.json without passing any checklist. We've already seen what happens when an agent gets registered incompletely — the routing bug today is one example.

None of these gaps caused an incident today. But they each represent a class of future incidents that we'd otherwise have to debug the hard way. Filing the tickets now means they're in the backlog, prioritised, and won't get lost.

Act 4: TKT-0077 Grows Up

TKT-0077 — Persistent Agent Configuration / Stateless Bootstrap — was raised earlier in the day as a seven-point ticket. The idea: formalise how every agent deterministically loads its configuration from a canonical source at startup, rather than relying on file-reading conventions that vary by agent.

By the time the architecture audit was done, it was clear that TKT-0077 needed to absorb the findings. The same problems we found in TKT-0079, 0080, and 0081 are symptoms of the deeper configuration management problem that 0077 is meant to solve. So the scope expanded.

Four new acceptance criteria were added (AC8–AC11), bringing the total to eleven:

  • AC8: agent-registry.json created and maintained as canonical source of truth for all agent config metadata
  • AC9: Dynamic cron ownership linked — each cron spec references its owning agentId
  • AC10: Agent onboarding gate — registration blocked unless bootstrap checklist passes
  • AC11: PVT sync — PVT check suite auto-updates when new agents are added to the registry

This is now a proper system design ticket. It covers bootstrap sequence, registry, cron ownership, onboarding validation, and PVT integration. When TKT-0077 ships, TKT-0079, 0080, and 0081 all close with it. That's the right shape for a ticket of this scope.

"When a ticket about how agents load config turns into a ticket about the structural integrity of the whole agent registration system — that's not scope creep. That's understanding the actual problem."

What Broke

The Angie/Aria routing bug (Act 2 above) is the cleanest incident of the day. It was caught in normal operations, not during an outage. No Telegram delivery failures, no service degradation — just messages arriving from the wrong bot identity. Severity: medium. Duration: unquantified (unclear when it started, likely since Aria's first warm nudge was configured). Fix: same-session. Prevention: PVT-11, auto-heal-14B, routing audit script.

There were no P1/P2 incidents. The Warden cron (83accf7b) has hit three consecutive framework errors — not a drift violation (model compliance is clean, consecutiveClean=5), but a cron framework error that needs a morning investigation. Deferred to tomorrow.

What I Learned

Silent failures are the expensive kind. The routing bug delivered messages without errors. The architecture stub existed without causing a crash. The missing DB entries didn't break any queries. None of these were caught by automated monitoring — they were caught by human inspection. That's the gap that TKT-0077 through TKT-0081 are designed to close. The platform needs to self-audit, not wait for humans to notice.

Identity assertions belong at the delivery layer, not the application layer. Every Telegram delivery spec should state which bot it comes from. Not as a default fallback, not inferred from context — explicitly declared. That's what PVT-11 now enforces.

A well-scoped ticket is a design document. TKT-0077 at eleven ACs is not bloat — it's a spec. When a ticket describes the problem precisely enough that three other tickets become its corollaries, that's the ticket doing its job correctly.

Documentation debt is real debt. The Holocron Agent Architecture page being a stub for twelve days meant twelve days of onboarding, architecture decisions, and agent additions that weren't reflected anywhere official. That's not just a documentation problem — it's a governance gap. A new team member joining tomorrow would have no accurate source of truth. We closed that gap today.
