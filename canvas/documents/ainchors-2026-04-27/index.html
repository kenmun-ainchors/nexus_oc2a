AInchors — Day 3 Build Log

From Reactive to Proactive: Building an AI Ops Resiliency Stack

One day. Two outage incidents. Thirty-nine changes. A full resiliency stack, a second AI agent, a governance layer, and a system that now heals itself every night at 23:30.

Ken Mun — CTO, AInchors · 27 April 2026 · Day 3
Opening

I came online to find the system dead.

Sunday morning, 6:15 AM. I opened the laptop and the AI platform was offline. The credits had run out overnight — Anthropic billing exhausted, fallback chain failed, Ollama authentication not configured. The system had silently stopped working sometime during the night.

This was Day 3 of building AInchors, and it started with two fires at once: an overnight outage that revealed gaps in the resiliency design, and a Day 2 journal that had been written in the wrong format — summary style instead of the verbatim-prompt record I'd established on Day 1.

By the end of the day, I'd rebuilt both. More importantly, I'd built the infrastructure to make sure neither type of failure could repeat itself. This is how Day 3 went.

Act I — 06:15 AEST

The Overnight Outage: A Chain of Compounding Failures

Incident Night-2 Billing + Auth Cascade Failure

Anthropic credits exhausted → fallback attempted Opus (same billing block, also failed) → fallback attempted Gemma4/Ollama → "No API key found for provider ollama" → system hard stop.

Root cause: Two gaps: (1) openclaw.json had a literal placeholder string "OLLAMA_API_KEY" — not a real key. (2) auth-profiles.json had no ollama:default entry at time of outage.

The fix was straightforward once I understood it. Both auth files patched, Ollama key set to ollama-local, and the backup cron refactored to be LLM-independent — dual-path with a systemEvent fallback so it fires even if the main session is down. Three Telegram cron routes that had been pointing to the wrong chat ID were also corrected.

But the more important fix was architectural: I filed US23 (resilient outage handling with automated fallback chain) and built the foundation for what would become the platform's resiliency stack. This incident was the catalyst for everything else that happened on Day 3.

Act II — 06:15–06:46 AEST

Setting the Record Straight: Locking the Journal and Blog Formats

The Day 2 journal had been written in high-level summary style. I rejected it. The standard I'd set on Day 1 was verbatim prompts — my actual words, timestamp by timestamp, nothing paraphrased away. Summary-style journals lose all the primary source context that makes them useful for auditing decisions, training future agents, and reconstructing intent weeks later.

A sub-agent rebuilt the Day 2 journal from source transcripts: 1,011 lines, 56 timestamped sections, 70 verbatim prompts. I reviewed and approved it at 06:44.

Then I locked the formats permanently.

🔒 Two distinct end-of-day artefacts Journal (memory/journal-YYYY-MM-DD.md) — Private. Yoda's voice recording my exact words. Chronological. Verbatim prompts → interpretation → actions → outcomes. This is the primary source of truth.

Blog (canvas/documents/ainchors-YYYY-MM-DD/index.html) — Public-ready. My first-person narrative. Reads from the journal, not the raw transcripts. Curated, coherent, stripped of sensitive values.

The rule: the blog is cooked from the journal. The journal is raw. You can't write the blog without the journal existing first.

Act III — 07:32–09:25 AEST

Building the Resiliency Stack: Three New Levels

At 07:32, I noticed something in the chat stats: the main agent had been running on claude-opus-4-7 — not Sonnet. An agent-specific config override had been set during early setup and never corrected. The entire morning had been running at ~3× the intended cost. That was the model drift incident.

One line change to revert it. But it revealed a pattern: the system had no mechanism to detect when critical configuration silently drifted from what had been approved. I needed to fix the pattern, not just the instance.

The result was three new levels of the resiliency stack, built over the next two hours:

The critical config baseline was the key anti-drift mechanism: a JSON file defining seven guarded configuration values, validated nightly by auto-heal Check #12. Any drift from baseline files a Notion user story for Ken's review. The Opus incident made that check permanent.

The pattern I'm building: every incident becomes a new auto-heal check. The system gets harder to break with every failure.

I also documented this in an evergreen ResiliencyFramework.md — a living reference covering design principles, all five levels, the incident-to-check feedback loop, and the roadmap toward OC2. That document will keep growing as the platform does.

Act IV — 09:27–09:49 AEST

ITSM Discipline: Ticket-First and Formalised Slash Commands

Good operations requires a ticket for everything. No ad-hoc work without a tracking reference. I stood up a proper ticketing system — scripts/ticket.sh with new/list/show/update/link/close commands, backed by state/tickets.json and mirrored to a Notion Service Tickets database.

The rule is non-negotiable: any task or request without an existing INC, US, or CHG reference gets a TKT raised before work starts. The first ticket raised — TKT-0001 — was the ITIL gap analysis itself. The system bootstrapped its own discipline.

I also formalised three slash commands as proper triggers with defined procedures:

Act V — 10:34–12:16 AEST

Aria: A Second Agent for the Business Stream

Up to Day 3, the platform had been entirely ops-focused. But AInchors has a business dimension too — and that needs its own AI agent, with its own rules, its own workspace, and direct access for the business lead.

I stood up Aria 🔵 as the Business Lead Agent. Isolated workspace (workspace-business/). CEO-level read access to all AInchors data. Write access scoped to business stream and the shared knowledge bridge. This is the OC2 precursor — Aria lives on OC1 temporarily and migrates when the dedicated hardware arrives.

The nightly context sync at 23:00 AEST keeps Aria informed of operational developments without creating a dependency loop. My morning standup now includes an Aria Oversight section — I surface her prior-day activity and flag anything needing attention.

But business stream output can't go external without review. That's what the governance layer is for:

🏛️ Governance Layer — Three Agents Shield 🔐 (Security, Sonnet) — data exposure, auth risks, policy violations
Lex ⚖️ (Legal, Opus) — contractual, regulatory, liability
Sage 🧪 (QA, Sonnet) — accuracy, completeness, quality

Any Aria output that goes external flows through scripts/governance-review.sh first. Veto on any tier triggers escalation.

I also built out the onboarding journey for the business lead — six structured stages covering orientation, platform access, project management (12 dedicated items: WBS, risk register, brief templates, stakeholder comms), business intelligence, external execution, and autonomy. A checklist JSON tracks progress.

Act VI — 19:30–21:55 AEST

What Broke (And What I Built to Stop It Happening Again)

Incident Gateway Crash — Bonjour/ciao Plugin Crash Loop

~19:30 AEST: Gateway crashed. Hard reset required. New session, no transcript continuity.

Root cause: Bonjour/ciao plugin crash loop — unhandled promise rejection firing every ~9 seconds, likely destabilising the gateway for some time before the fatal crash.

Fix: Bonjour plugin disabled. Gateway restarted cleanly. Context fully recovered from files in ~10 minutes.

The crash was actually a validation of something I'd built: the entire operational context was in files. MEMORY.md, the daily log, the CHANGELOG, state files. Recovering from a crash wasn't about session state — it was about having a complete written record. That's the principle I keep coming back to: files are the recovery mechanism.

The second problem was Gemma4. I'd set it as Aria's default model and extended it to all business stream agents. The idea was cost savings — Gemma4 runs locally at zero API cost. The reality: a 26B model cold-loading from disk on every Aria Telegram message caused 30–60 second delays and degraded my own webchat in the process.

⚠️ Lesson: Gemma4 = background only, forever Interactive traffic + large local models = bad UX. Gemma4 is now reserved exclusively for: backup crons, cost tracker, and non-interactive batch work. Never in the user-facing path.

The Telegram routing architecture was the third problem. With a single bot serving two agents, stale session state for one agent's user would persist through restarts and cause misrouting. I tried configuration fixes. They kept failing.

The clean solution: two bots, one per agent. @AInchorsOC1Bot → Yoda (my channel). @AInchorsAriaBot → Aria (business channel). Strict allowlists. No shared state. Problem solved architecturally, not with workarounds.

Lessons

What I Learned on Day 3

  1. Files are the recovery mechanism. Gateway crashed. Context restored in 10 minutes from files. Invest in files, not in trusting session state.
  2. Config drift is the stealth enemy. Two Opus drift incidents in one day — both from config overrides that survived restarts. Critical config baseline + nightly validation is mandatory, not optional.
  3. The architectural fix beats the config fix. Multiple attempts to fix Telegram routing through config all failed. Splitting into two bots solved it immediately. When a workaround keeps breaking, that's the system telling you the architecture needs to change.
  4. Ticket-first catches scope creep early. Raising TKT-0001 against the ITIL work itself forced me to scope and estimate before diving in. The discipline pays off immediately.
  5. Every incident should make the next one harder. That's the feedback loop I'm building into auto-heal: failure → check → automated prevention. The platform compounds on its own reliability.
