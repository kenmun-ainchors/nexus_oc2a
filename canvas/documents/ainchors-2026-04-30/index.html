AInchors Build Log · Day 6 · Thursday 1 May 2026

The Platform Learns to Watch Itself

Four user stories delivered, two crashes traced to a 17KB file, and the day the observability layer turned its gaze inward.

Ken Mun — CTO, AInchors 30 April 2026 ~2,400 words

Day 6 was supposed to be about observability. I had four goals walking in: build a logging sub-agent, build a task monitor, formalise the model cost tier framework, and introduce a new delivery model. All four got done. But the most important thing that happened today wasn't on my list.

We built an observability system for our AI platform — and then the platform crashed twice. And our brand-new observability system didn't catch it. That gap, and how we closed it, is the real story of Day 6.

Act 1 — Clearing the Decks

I came in off the back of Day 5 infrastructure work with a few loose ends. The Aria relay poller was throwing 8 consecutive errors — root cause was a prompt that told the agent to produce no output. OpenClaw can't parse an empty response, so it treated silence as failure. Fixed in 15 minutes: force a single status line output. Clean run confirmed, 0 errors.

I also closed US18, the SLA report. The numbers weren't flattering: 97.46% availability against a 99% target, and 66-minute MTTR against a 60-minute target. But the value of having the report isn't the pass/fail — it's that all five recommendations from it are now actioned. Root causes fixed, documentation rebuilt, detection intervals tightened. US39 (preventable downtime) raised as a future story and parked properly.

Act 2 — The Sprint: Four Stories, One Morning

US40: Observability Logging

The brief was simple: build a lightweight logging layer that captures failures, errors, and anomalies — feeding the morning standup, SLA reports, and continuous improvement. No heavy tools. SQLite. 7-day rolling window.

What shipped: obs-init.sh, obs-log.sh, obs-query.sh, obs-collector.sh, and state/obs.db with four check types (A–D). Health-check, auto-heal, and SLA report scripts all patched to write to the database. First confirmed run: all four checks passing, database populated.

One early finding: the cron I built used Haiku (our Tier 1 model) for pure script execution. Wrong choice — script runs don't need an LLM at all. That led directly into US42.

US42: The 3-Tier Cron Framework

Ollama routing doesn't work inside OpenClaw's isolated agentTurn crons. And Gemma4:e2b's context window is 2,048 tokens — well below the platform minimum of 16,000. So "use Gemma4 for cheap crons" isn't currently viable. That surfaced a more important question: which crons need an LLM at all?

Decision: 3-Tier LLM Classification

Tier 0 (No LLM): Pure exec — health checks, log collectors, task monitors. Use systemEvent.
Tier 1 (Barely needs LLM): Simple conditional output — Gemma4:e2b when routing is fixed, Haiku fallback.
Tier 2 (Needs LLM): Reasoning or content generation — Haiku → Sonnet.

Gate rule: classify every new cron as Tier 0/1/2 before creating it.

I converted four crons — health-check, obs-collector, task-monitor, relay-poller — from agentTurn (Haiku) to systemEvent (no LLM). Estimated saving: ~$1.50/day. The framework is now locked in Agents/ModelStrategy.md.

US41: Task Monitoring

I want to know when a sub-agent stalls. Simple requirement: always-on lightweight tracker, alert on failure, silent on success. Stats feed the morning standup.

What shipped: scripts/task-collector.sh feeding state/tasks.db, integrated with the heartbeat watchdog. If a task stalls, state/task-stall-alert.json is written and Yoda picks it up within 30 minutes. Option C hybrid design: explicit registration on spawn plus auto-discovery safety net.

US43: RTB Operating Model

This was the operating model shift. With our core frameworks all at maturity level 2 or 3, the platform can run. Ken's instruction: pivot focus toward business, and change how we decide what to work on each day.

Before today, I proposed a sprint based on whatever was next in the backlog. Starting tomorrow, I use RTB — Rose, Thorn, Bud:

  • Rose: What new opportunity appeared in yesterday's data?
  • Thorn: What should we stop doing?
  • Bud: What can we do better?

Two items per stream (technical + business), drawn from obs.db and Aria's daily brief. If no data signals a recommendation, I fall back to the next priority in the backlog. One constraint: no framework can outpace the others by more than one maturity level. The pacing rule is enforced — if any framework races ahead, the BUD recommendation targets the laggard.

First RTB delivery: Friday 1 May, 8AM standup.

Act 3 — The Crash We Couldn't See Coming

Incident · INC-20260430-1006-A and 1006-B

Two WebSocket 1006 gateway crashes. 12:11 AEST and 18:11 AEST. Both from the same root cause. The second one happened after we started investigating the first.

What Happened

Aria's SOUL.md was 17,393 characters. OpenClaw silently truncates SOUL.md files at approximately 10,000 characters on load. Silently — no warning, no error, no log line.

The truncated file was missing the Telegram target IDs near the end. So Aria was routing messages to the wrong bot. Wrong targets meant messages that would never receive a response. That stuck the agent session. A stuck session accumulates unreleased context. Enough of those and the gateway hits OOM. OOM → gateway crash → WebSocket 1006.

Two crashes. Both traceable back to one oversized file. The fix took 15 minutes: compact Aria's SOUL.md from 17,393 chars to 3,765 chars. All detailed procedures moved to ARIA_RULES.md.

The Monitoring Gap

Here's the part that stung: I asked whether our new obs-collector had captured the error. It hadn't. The observability layer I'd built that morning was watching four things — health, API connectivity, cron failures, disk space. It wasn't watching gateway.err.log.

We built observability for our agents. We forgot to build observability for the platform running the agents.

What We Fixed

obs-collector expanded from 4 checks (A–D) to 14 checks (A–N), plus 13 gateway error patterns including 1006, OOM, session-stuck, and failed relay attempts. Check K specifically monitors SOUL.md size — alert fires at 6,000 characters, before we hit the truncation threshold.

Total platform monitoring coverage now: gateway errors, agent session OOM, cron failures, state file corruption, Warden escalation queue, task stalls, SOUL.md size, disk space, cost state staleness, API connectivity. If it can fail, obs-collector now knows about it.

Act 4 — The Rule That Can't Break

After fixing Aria, I checked the other agents. Sage: 5,463 chars. Lex: 5,974 chars. Both approaching the soft threshold. Both compacted. All six agents now have:

  • SOUL.md — compact identity, traits, brief rules, cadences. Under 5,000 chars.
  • [AGENT]_RULES.md — all detailed procedures, protocols, and flows.
Non-Negotiable Standard — Locked 30 April 2026

Every agent SOUL.md must stay under 5,000 characters. Hard limit: 10,000 chars (OpenClaw truncation threshold). New agents: SOUL.md written compact from Day 1. Auto-compact autonomy granted to Yoda — no Ken approval needed when any file hits 6,000 chars.

This is now a platform architectural rule, not a one-off cleanup. The lesson is locked in MEMORY.md. The watcher (obs-collector Check K) is watching for it. The rule exists because the failure is silent, remote, and catastrophic — and by the time you see the 1006, you've already lost the thread.

What I Learned

1. Silence is the worst failure mode.

OpenClaw truncating a SOUL.md file produces no error, no warning, nothing. The crash happens three steps removed from the cause. Any system that fails silently will fail at the worst possible time. The fix isn't just compacting the file — it's building a detector that catches the condition before it becomes a crash.

2. Your observability system needs its own observability.

I built an obs-collector that morning and assumed it was comprehensive. It wasn't. It was monitoring the agents but not the platform they run on. Every monitoring layer has a blind spot — find it before an incident does.

3. Classify the LLM cost before you build.

The 3-tier cron framework isn't about saving $1.50/day. It's about not burning LLM tokens on work that a shell script can do. The gate rule — classify before you create — forces the right question at the right moment. That discipline compounds.

4. Operating model shifts need a clear forcing function.

RTB isn't just a standup format change. It's a signal that the build phase is maturing. When frameworks reach level 2-3, you stop building foundations and start running operations. The model shift makes that explicit and measurable.
