RTB: What We Learned From Week 1

Every week we run an RTB — Rose, Thorn, Bud. It's a fast retrospective format that forces honest evaluation before we move forward.

Rose: The LinkedIn metrics pipeline is live. Scripts capturing engagement data at each interval. The system can now learn what works.

Thorn: A nasty one. Our auto-heal system had been logging routine git commits as incident records. Seven false incidents in the log. Architecture Clarity: Who Decides What

One of the harder problems in a multi-agent system is decision authority. Who makes which call? Yesterday Atlas was trying to do enterprise architecture AND platform-internal decisions. That's too broad — EA should stay at the business and technology architecture level.

Today I restructured it:

  • Atlas handles enterprise-level: business architecture, technology roadmap, integration strategy, regulatory alignment
  • Thrawn handles platform-internal: agent orchestration, model selection, S1–S7 security controls, observability design
  • Yoda orchestrates: routes architectural questions to the right agent, synthesises cross-cutting decisions

Clarity in roles matters. Especially when the "people" making decisions are AI agents. Ambiguity in scope leads to agents either duplicating work or leaving gaps.

The Gateway Incident That Almost Wasn't

At 18:45 I ran a health check. The event loop was at 93.8% saturation. Sessions were showing 28-second delays. The platform was close to self-inflicted failure.

Root cause: two task runs from May 1 were still flagged as "running" in the system — zombie tasks that were blocking the event loop but hadn't been cleaned up. Four days of accumulated ghost state.

INCIDENT Gateway event loop at 93.8% saturation. Two zombie task runs from May 1 blocking the event loop. Also: invalid tool configurations in agent allowlists consuming unnecessary resources. Both cancelled, allowlists cleaned. Loop cleared immediately.

This was educational. Three observability gaps were exposed:

  • auto-heal had no zombie task detection
  • health monitoring had no event loop saturation check
  • the task watchdog only monitored workspace tasks, not runtime tasks

All three fixed same-day. The /stabcheck command is now available for on-demand platform health audits.

The principle I'm applying: every incident should permanently improve the system. Not just fix the symptom. Fix the detection gap that let it get this far.

Krennic: The SRE Agent We Need Before the First Client

The incident today made a clear case for a dedicated SRE agent. Right now when something breaks, Yoda investigates. That's fine when I'm the only "client." It won't be fine when I have paying customers.

Krennic — named after Director Orson Krennic, the meticulous Death Star project director — will own: incident response, SLO definitions, error budgets, runbooks, post-mortems, and capacity planning. The brief I gave: reliability engineering as a first-class discipline, not an afterthought.

"We at AInchors don't have dark/light force differentiation. It's the principles, dedication and commitment that we align and value."

That quote came up in context of naming an agent after a character who builds a superweapon. The point stands: the name is about the character's attributes — methodical, detail-obsessed, project-completion focused — not their moral alignment in a fictional universe. We take reliability seriously.

Security: Least Privilege Applied

S4 from the security charter — per-agent tool scopes. Every agent now only has access to the tools it needs for its specific role.

The governance agents (Shield, Lex, Sage, Warden) have read-only filesystem access. No execution permissions. Spark has social and content tools, not system access. The principle: an agent that can't do harm, can't accidentally do harm.

This is basic security hygiene. It took most of the morning to apply correctly across all 9 active agents. Worth it.
